gcp-user-access-management-service — Comprehensive Technical Reference
Executive Summary
gcp-user-access-management-service is a Spring Boot 3.5 / Java 17 REST API (v1.5.5) that serves as the central access control and user lifecycle management hub for the WDP (Worldpay Data Platform) NAP (Network Access Platform) ecosystem. It has four primary responsibilities: (1) managing a two-level merchant entity hierarchy (Parent → Child entities), (2) managing Merchant IDs (MIDs) and their relationships to that hierarchy, (3) provisioning and managing users in an external SunGard Identity Provider (IdP), and (4) authorising inbound JWT-bearing requests by cross-checking JWT claims against the merchant hierarchy. All inbound calls arrive as HTTPS/REST on port 8082 behind a Kubernetes nginx Ingress; all callers must present an OAuth2 Bearer JWT. The service makes outbound calls to two SunGard IdP instances and to AWS S3 for bulk file uploads, and reads/writes two PostgreSQL database schemas. It does not issue JWTs, store credentials, process payments, manage cases, or send emails directly.

Part 1: Primary Responsibility
What the Service Does
1. Merchant Hierarchy (Entity) Management
The service manages a two-level parent/child entity hierarchy in the nap PostgreSQL schema:1
JPA EntityDB TableDescriptionParentEntitynap.nap_parent_entityTop-level grouping (e.g., merchant organisation)ChildEntitynap.nap_child_entitySub-groupings beneath a parentMerchantEntitynap.nap_merchantIndividual Merchant IDs with MCC and WPG IDMerchantRelationshipEntitynap.nap_entity_relLinks merchants to parent/child entitiesCaseEntitynap.caseDispute case → merchant ID mapping (read-only by this service)
When a parent entity is created, a matching default child entity of the same name is automatically created.2
2. Merchant (MID) Management
Merchants are stored in nap.nap_merchant with fields: merchantId, merchantName, mcc (Merchant Category Code), wpgId (Worldpay Gateway ID). The service supports creating, updating, deleting, and searching merchants, always linked to a parent/child entity.3
3. User Lifecycle Management via IdP
The service acts as a proxy/orchestrator to a SunGard IdP, supporting two distinct "firms":4
ConstantValueIdP Config KeyUS_MERCHANT_FIRM"US_Merchant"idp.user-api-base-urlMERCHANT_DISPUTES_FIRM"Merchant_Fraud_Disputes"idp.mfd-user-api-base-url
Operations: create user, update user, get user, list users, reset password, activate/deactivate/suspend/resume. No user credentials are stored — all are delegated to the IdP.
4. ACL (Access Control List) Management
ACL entries in wdp.acl link named consumers to source systems (CORE, VAP, NAP) and entity values.5 Fields: consumerName, sourceSystem, entityType, entityValue, status, with full audit timestamps (createdBy/activatedBy/deactivatedBy).
5. JWT-Based Runtime Authorisation
A dedicated POST /authorize endpoint validates whether a JWT-bearer's napParentEntity/napChildEntities claims grant access to a given merchant ID or dispute case number, by cross-referencing nap.nap_entity_rel.6

Identified by: Any valid JWT

Data Ownership, /authorize, CRUD vs Auth, and Access Control — Deep Dive
Executive Summary
This service owns and persists two distinct data domains: (1) a merchant entity hierarchy + MID relationships in the nap PostgreSQL schema, and (2) an access control list (ACL) in the wdp schema. The /authorize endpoint performs a runtime read-only check — it looks up a merchant's parent/child entity relationships and validates them against JWT claims; it does not grant or deny access to any resource, just validates that a JWT-bearer "can see" a given merchant or case. This service is both an authorization-check service AND a full CRUD service — it manages the reference data (entities, merchants, users, ACLs) that authorization decisions are based on. Access control is enforced in two layers: a transport-level JWT validation (Spring OAuth2 Resource Server), and an application-level scope enforcement that extracts napParentEntity/napChildEntities claims from the JWT and restricts every data query and mutation to only the entities the caller has been granted.

What It Looks Up
The logic is in AuthorizationServiceImpl.authorizeEntity() and authorizeNAPToken():14
Step 1: Check JWT issuer
  └─ if iss contains "us_worldpay_fis_int" → return true immediately (always authorized)

Step 2: Extract JWT claims
  ├─ napParentEntity  (required; throws UnauthorizedException if blank)
  └─ napChildEntities (optional; comma-separated child IDs)

Step 3: Resolve merchant ID
  ├─ if caseNumber provided → query nap.case WHERE I_CASE = caseNumber → get C_LEVEL1_ENTITY (merchantId)
  │    └─ throws UnauthorizedException if case not found or has no merchantId
  └─ else → use request.merchantId directly

Step 4: Authorisation check (depends on JWT shape)

  Case A: napChildEntities present AND napChildEntities.size() ≤ 4
    → native query against nap.nap_entity_rel:
      SELECT ... FROM nap.nap_entity_rel WHERE
        i_mid = ? AND i_parent_entity_id = ? AND i_child_entity_id = ?
        (one clause per child entity, joined with OR)
    → if rows found → return true
    → else → throw UnauthorizedException

  Case B: napChildEntities present AND size > 4  (falls through to Case C)

  Case C: napChildEntities absent (parent-only user)
    → query MerchantRelationshipEntityRepository.findByMerchantId(merchantId)
      SELECT i_parent_entity_id FROM nap.nap_entity_rel WHERE i_mid = merchantId
    → if no rows → throw UnauthorizedException
    → check if any row's parentEntityId matches napParentEntity (string comparison)
    → if no match → throw UnauthorizedException
    → if match → return true

  Case D: Both napChildEntities present AND size > 4
    → query MerchantRelationshipEntityRepository.findByMerchantId(merchantId)
    → check if any row's childEntityId is in the JWT's napChildEntitiesSet
    → if none match → throw UnauthorizedException
    → if match → return true
What It Returns
Success (200 OK): Empty body. The HTTP 200 itself is the signal that the JWT-bearer is authorised for the requested merchant/case.12
Failure: One of:

401 Unauthorized — JWT claims don't cover the merchant
400 Bad Request — missing/invalid platform, missing both caseNumber and merchantId, case not found

The endpoint does not return any entity data, permission sets, or user information. It is a pure boolean gate expressed as HTTP status.

Part 4: How Access Control Is Enforced
Access control is implemented in two distinct layers.
Layer 1 — Spring Security (Transport-Level JWT Validation)
Handled by SecurityConfig.15 Every request to a business endpoint must present a valid Bearer JWT. Spring validates the token's signature and expiry against one of the configured trustedIssuers. No application code runs if this fails — Spring returns 401 before the controller is reached.
Layer 2 — Application-Level Scope Enforcement
Once the JWT is valid, the application reads custom claims from it and applies business rules. This is entirely in AuthorizationUtil and the controllers/services. There are four distinct rule categories:
Rule 1: Internal vs. External Split (applies to ALL endpoints)
java// AuthorizationUtil.java:51-62
Boolean isInternalFirm = AuthorizationUtil.validateInternalFirm(jwt);
// Reads jwt.claims["iss"]
// Returns true if iss contains "us_worldpay_fis_int"

Internal callers (isInternalFirm = true): Bypass ALL entity-scoping, ALL role checks. Full access.
External callers (isInternalFirm = false): Subject to all rules below.16

Rule 2: Role-Based Action Gating (user management endpoints)
For external callers on user management endpoints, AuthorizationUtil.validateUserRoles() or validateNapUserRoles() checks the AuthorizationList JWT claim:17

Pods, Scaling, Memory Configuration & Architectural Risks
gcp-user-access-management-service

Executive Summary
The pod replica count is a template placeholder ({{ replicas-user-access-management-service }}) resolved by XL Deploy at deploy time — the actual number is not visible in the codebase. There is no Horizontal Pod Autoscaler (HPA): scaling is purely static. Memory is the only resource with configured limits (2 GB limit / 1 GB request); CPU is completely unset. The codebase contains several architectural concerns ranging from a confirmed NPE bug in updateEntity, through a topology spread constraint that can never take effect (wrong label), to no connection/read timeouts on outbound HTTP calls and an S3 region hardcoded to EU_WEST_2 regardless of deployment environment.

Pod Count & Replica Configuration
Static Replica Count (Placeholder)
yaml# resources.yaml:8
spec:
  replicas: {{ replicas-user-access-management-service }}
The value {{ replicas-user-access-management-service }} is an XL Deploy template placeholder1. It is substituted at deployment time using the deployit-manifest.xml configuration2, which references an environment-specific variable store managed by the XL Deploy/DeployIt system. The codebase does not contain a default value. The actual replica count per environment (dev / staging / prod) is stored externally in the XL Deploy variable dictionary and is not visible in this repository.
Rolling Update Strategy
yaml# resources.yaml:9-13
strategy:
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
  type: RollingUpdate

maxSurge: 1 → during a deploy, at most N + 1 pods run simultaneously3
maxUnavailable: 0 → no pods are terminated until a replacement is ready
minReadySeconds: 30 → a new pod must be healthy for 30 seconds before the old one is removed4

No HPA
There is no HorizontalPodAutoscaler resource anywhere in resources.yaml or the wider codebase5. The replica count is static; the service cannot auto-scale in response to CPU, memory pressure, or custom metrics.

At default settings, each pod holds up to 20 database connections (2 pools × 10). Without explicit sizing this could be either over- or under-provisioned depending on database server limits.

Architectural Concerns & Known Risks
RISK-1 (HIGH): NPE / Incorrect Guard Order in updateEntity
java// AccessManagementServiceImpl.java:129-132
Optional<ParentEntity> entity = accessManagementDao.getParentEntity(entityId);
String childName = entity.get().getParentName();   // ← line 130: called BEFORE isPresent() check
if (!entity.isPresent()) {                          // ← line 131: too late
    throw new BadRequestException(...);
}
entity.get() is called unconditionally at line 130 before the isPresent() guard at line 131. If no parent entity exists for entityId, this throws NoSuchElementException (an unchecked exception) rather than the intended BadRequestException. Callers receive an unhandled 500 rather than a descriptive 40013.
RISK-2 (HIGH): Topology Spread Constraint Uses Wrong Label
yaml# resources.yaml:25-31
topologySpreadConstraints:
  - maxSkew: 1
    whenUnsatisfiable: ScheduleAnyway
    topologyKey: kubernetes.io/hostname
    labelSelector:
      matchLabels:
        app: gcp-user-access-management-service${BRANCH_NAME_PLACEHOLDER}  # ← "gcp-" prefix
But the pod template labels are:
yaml# resources.yaml:6, 16, 20
metadata:
  labels:
    app: user-access-management-service${BRANCH_NAME_PLACEHOLDER}  # ← no "gcp-" prefix
The labelSelector references gcp-user-access-management-service while the pods are labelled user-access-management-service. The constraint never matches any pod. whenUnsatisfiable: ScheduleAnyway means Kubernetes will silently schedule all pods to the same node with no warning14.
RISK-3 (MEDIUM): No Timeouts on Outbound HTTP (RestTemplate)
java// CommonConfig.java:11-13
public RestTemplate getRestTemplate() {
    return new RestTemplate();
}
The RestTemplate bean is created with default JDK HttpURLConnection and no timeouts15. All IdP calls (IdpRestInvoker.java) use this bean16. If the SunGard IdP is slow or unreachable, request threads will block indefinitely, exhausting Tomcat's thread pool and rendering the service unresponsive to all other callers.
RISK-4 (MEDIUM): S3 Region Hardcoded to eu-west-2
java// S3ClientConfig.java:18
s3Client = S3Client.builder().region(Region.EU_WEST_2).build();
The AWS region is hardcoded to EU_WEST_217. The service is deployed from ECR us-east-2 (784102274315.dkr.ecr.us-east-2.amazonaws.com/frauddispute/...)18. If the S3 bucket is in a different region (e.g., us-east-2 matching the deployment), all S3 upload requests for bulk file upload will be cross-region, increasing latency and potentially failing if the IAM policy or bucket policy is region-locked. The region is not configurable via environment variable.
RISK-5 (MEDIUM): @Async Without Explicit Executor Qualifier
java// AccessManagementDaoImpl.java:178, 195
@Async
public CompletableFuture<List<ChildEntity>> searchChildEntitiesByParentEntityIds(...)
@Async
public CompletableFuture<List<MerchantRelationshipEntity>> searchMerchantEntitiesByParentEntityIds(...)
The @Async annotation does not specify "asyncExecutor". Spring resolves unqualified @Async by looking for a bean named taskExecutor; failing that it falls back to SimpleAsyncTaskExecutor (which creates a new thread per call, unbounded)19. The configured pool bean is named asyncExecutor20, not taskExecutor. Whether the custom pool is actually used depends on whether it is the sole Executor/TaskExecutor bean at runtime. If Spring Boot registers additional executors (e.g., for scheduling), the custom pool may be silently bypassed. The safe fix is @Async("asyncExecutor").
RISK-6 (MEDIUM): No CPU Limits or Requests
resources.yaml specifies only memory limits and requests; CPU is completely absent21. Consequences:

No CPU request → Kubernetes scheduler cannot bin-pack pods by CPU; all pods may land on the same node
No CPU limit → a CPU-intensive request (large bulk upload, heavy pagination) can monopolize node CPU at the expense of other workloads
Combined with no HPA, there is no automated response to CPU pressure

RISK-7 (LOW): Platform Stubs in Production Code
java// AuthorizationServiceImpl.java — non-NAP platform handling
case CORE: case VAP: case LATAM: case PIN:
    throw new BadRequestException("Implementation is in progress");
POST /authorize for platforms CORE, VAP, LATAM, and PIN unconditionally returns 400 "Implementation is in progress"22. This stub code is deployed to production. Any caller attempting to authorize against these platforms gets a 400 with a developer-facing message rather than a more appropriate 501 Not Implemented or feature-flag-controlled response.
RISK-8 (LOW): In-Memory Count Pattern — Potential Memory Pressure
getChildDetails() in AccessManagementServiceImpl.java (lines 265-277) loads all MerchantRelationshipEntity records for a page of child entity IDs via a single unbounded JPA query, then counts by streaming/filtering in Java23. For merchants with thousands of MIDs across many children, this pattern loads large result sets into JVM heap rather than delegating the aggregation to the database. A COUNT(*)... GROUP BY child_entity_id query would be correct; this pattern is also present in the getParentDetailsWithMerchantCount and getParentDetailsWithChildCount sort paths (lines 413, 470).
RISK-9 (LOW): @Transactional Without Manager on AclServiceImpl
java// AclServiceImpl.java:46
@Transactional
public AclResponse createAcl(...)
AclServiceImpl writes to the wdp schema but uses plain @Transactional (no transactionManager qualifier)24. This works because wdpTransactionManager is annotated @Primary25, so Spring selects it by default. The risk is implicit: if @Primary is ever removed or the nap manager becomes primary, wdp writes silently use the wrong transaction manager and roll back incorrectly.
RISK-10 (LOW): Swagger / API Docs Exposed in All Non-Prod Environments
java// SecurityConfig.java:35-37
authWhiteList = new String[] {
    "/actuator/health", "/access-management-service-api-docs",
    "/swagger-ui/**", ... };
In non-production environments Swagger UI and the OpenAPI spec are on the auth whitelist26. Any network-reachable client can enumerate all 25 endpoints, their request contracts, and their error messages without a JWT. This is acceptable for dev/staging but could be a concern if non-prod hosts are accessible outside the corporate network.
