## Executive Summary

`wdp-chargeback-evidence-event-scheduler` is a **Spring Boot long-running service** (Kubernetes `Deployment`, not a `CronJob`) that acts as the **outbox relay layer** for the WDP chargeback and evidence domain. It owns five independent `@Scheduled` cron jobs (and other upstream writers) have already committed, it does not call FileProcessor, and its relationship to FileProcessor is purely through the shared database. The most critical scheduler (Scheduler1) enforces a sequencing dependency: `EVIDENCE_ATTACH` rows are held in `BLOCKED` status until their paired `CHARGEBACK_PROCESS` row reaches `SUCCESS`, preventing out-of-order evidence publication.

---

## 1. Primary Responsibility

### What It Does

The service is the **transactional outbox relay** for the WDP chargeback/evidence pipeline. Its five schedulers collectively:

1. **Scheduler1 (`ChargebackEvidenceEventScheduler`)** – polls `wdp.chbk_outbox_row`, publishes `CHARGEBACK_PROCESS` events to the `new-case-events` topic and `EVIDENCE_ATTACH` events to the `case-evidence-events` topic, then unblocks or error-marks dependent evidence rows.[^1]
2. **Scheduler2 (`FileJobStatusCompletionScheduler`)** – polls `wdp.file_job` for `PROCESSING` jobs, computes completion percentages from `wdp.chbk_outbox_row` counters, marks jobs `COMPLETED`, and archives old `SUCCESS` rows into `wdp.chbk_outbox_row_archive`.[^2]
3. **Scheduler3 (`OutgoingEventScheduler`)** – polls `wdp.outgoing_event_outbox` and routes events to topic(s) determined by `channelType` (e.g., `EXPIRY_EVENTS`, `CORE_EVENTS`, `GP_EVENTS`, `SEN_EVENTS`).[^3]
4. **Scheduler4 (`BreOrchestrationOutboxScheduler`)** – polls `wdp.bre_orchestration_outbox` and routes events to either the `business-rules` topic (`BUSINESS_RULES` component) or the `outgoing-events` topic (`NOTIFICATION_ORCHESTRATOR` component).[^4]
5. **Scheduler5 (`EvidenceErrorEmailScheduler`)** – polls `wdp.file_evidence` for ERROR rows updated yesterday, generates S3 pre-signed URLs, renders a CSV report, and calls an external email notification REST endpoint.[^5]

### What It Explicitly Does Not Do

- Does **not** consume from Kafka (no `@KafkaListener`; no consumer configuration present).
- Does **not** parse or validate incoming CSV/file uploads. File parsing is done by a separate FileProcessor component that writes to the outbox tables.
- Does **not** call the Worldpay chargeback API or any card-scheme endpoint directly.
- Does **not** perform schema migration (Hibernate DDL is set to `false`[^6]).
- Does **not** implement any HTTP REST API endpoints for external callers (no `@RestController`).

### Relationship to FileProcessor

The scheduler is **downstream and independent** of FileProcessor. FileProcessor (or other upstream writers) populates `wdp.chbk_outbox_row` and `wdp.file_job`. The scheduler only reads those tables; it never calls FileProcessor. The coupling is entirely through the shared PostgreSQL schema. Scheduler2 monitors `file_job` rows that are set to `PROCESSING` by FileProcessor and transitions them to `COMPLETED` once all corresponding outbox rows are terminal.[^2]

## 2. Trigger and Schedule

### Trigger Mechanism

All five schedulers use **Spring `@Scheduled(cron = ...)` annotation** – a simple time-based cron trigger running inside the Spring container's `ThreadPoolTaskScheduler`.[^7] There is no SQS trigger, no Kubernetes `CronJob`, and no event-driven mechanism.

### Scheduling Infrastructure

- `@EnableScheduling` is declared on the main application class.[^8]
- A custom `SchedulerConfig` bean creates a `ThreadPoolTaskScheduler` with **pool size 5** and thread name prefix `ChargebackEvidence`.[^9] This is exactly enough threads for all five schedulers to run concurrently if their fire times overlap.

### Cron Values

All cron expressions are **externalized via environment variables injected through Kubernetes Secrets / Spring profiles**. No literal cron expressions appear in the checked-in code. Property keys and their injection source:

### Kubernetes Resource Type

**`Deployment`**, as declared in `resources.yaml`.[^11] This is an always-running pod with a rolling-update strategy (`maxSurge: 1`, `maxUnavailable: 0`). It is **NOT** a Kubernetes `CronJob`.

**Query:**[^13]
```sql
SELECT outbox FROM ChbkOutboxEntity outbox
WHERE (outbox.status = 'PENDING')
   OR (outbox.status = 'FAILED' AND outbox.retryCount < 3 AND outbox.nextRetryAt <= CURRENT_TIMESTAMP)
   OR (outbox.status = 'PENDING_DEFERRED' AND outbox.nextRetryAt <= CURRENT_TIMESTAMP)
```

**Ordering and Page Size:**
- Sorted by `id` ASC.[^13]
- Page size from `app.scheduler.properties.pageSize` → env `${page_size}` (value not visible in committed config).[^14]
- Always fetches page 0 (`PageRequest.of(0, pageSize, Sort.by("id"))`), then loops with `hasMoreRecords = page.hasNext()` until all eligible rows are consumed.[^1]

**In-progress prevention:** The scheduler sets `status = 'PUBLISHED'` and **saves to the database before calling Kafka**.[^15] This optimistic lock-before-publish approach means: if the pod is killed after the save but before the Kafka send, the row remains in `PUBLISHED` status and is NOT requeued (it falls out of the `findEligibleOutboxRecords` filter). This is a deliberate design to prevent duplicate publish, at the cost of potential message loss on crash.

6. Error Handling
Retry Strategy on Kafka Publish Failure
All four Kafka-publishing schedulers (1, 3, 4, and implicitly Scheduler1 for evidence) share the same error-handling pattern:2
retryCount = entity.retryCount + 1
if retryCount > 2: → status = ERROR  (terminal, no further pickup)
else:              → status = FAILED, nextRetryAt = now + 3,600,000ms (1 hour)
This gives up to 3 total attempts (initial + 2 retries, since retryCount starts at 0 and becomes ERROR when > 2). The 1-hour retry delay is hardcoded as ApplicationConstants.NEXT_RETRY = 3600000L.[^34]
Producer-Level Retry (Kafka Client)
The KafkaProducerConfig sets:

RETRIES_CONFIG = ${spring.kafka.retry-count} (= 3 in all environments)[^35]
ENABLE_IDEMPOTENCE_CONFIG = true
ACKS_CONFIG = "all" (wait for all ISR replicas)
MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION = 5[^36]

So there are up to 3 Kafka-client-level retries per kafkaTemplate.send().get() call, before the application-level exception propagates upward.
What Happens to a Row That Cannot Be Published
After 3 application-level failures the row reaches status = ERROR. ERROR rows are not requeued by the scheduler queries (none of the findEligibleOutboxRecords queries include status = 'ERROR').3[^19][^21] The row persists in the table with errorMessage set. No automated DLQ or secondary retry mechanism exists in this codebase for ERROR rows.
Dead Letter Mechanism

No Kafka DLQ is configured or referenced anywhere in the codebase.
No error archive table for ERROR rows (archive only moves SUCCESS rows after 30 days[^37]).
Scheduler5 provides an indirect error visibility mechanism: it queries file_evidence.attachment_status = 'ERROR' and sends a daily email report with S3 pre-signed links to the failed documents, but this is alerting only – it does not reattempt publishing.

One Row Failure vs. Continue Processing
Each row is processed inside a for loop with a per-row try/catch. On exception, the catch block saves the updated error state and continues to the next row.415 One failing row does NOT stop the batch. The outer scheduler method also has a try/catch that logs the exception, so even an unexpected error in the outer loop does not crash the Spring context.

7. Idempotency
Idempotency Key
Every ChbkOutboxEntity has an idempotency_id UUID column.6 Before calling kafkaService.publishEventToKafka(), this UUID is passed as a Kafka message header idempotency-key.[^38] The consumer side is responsible for deduplication using this header; the scheduler itself does not enforce consumer-side idempotency.
At-Least-Once vs. Exactly-Once
The publisher has a mark-before-send pattern:

Set status = PUBLISHED → save to DB.
Call kafkaTemplate.send().get().

If the process dies between step 1 and step 2, the row is left in PUBLISHED and will never be retried (PUBLISHED is not in any eligible-query filter). This means:

If step 2 never executes: the message is lost (under-delivery).
If step 2 succeeds but the status update afterwards fails: step 2's Kafka metadata is not written back to the DB, but the message was still delivered.

The Kafka producer is configured with enable.idempotence=true and acks=all, which gives exactly-once delivery at the Kafka broker level for a single producer session.[^36] However, across pod restarts idempotency is not guaranteed because the producer sequence is re-initialized.
Summary: The design is at-most-once for the scheduler-to-Kafka hop (mark-before-send prevents replay on crash). Consumer-side deduplication via the idempotency-key header is the stated mechanism for downstream protection, but implementation is not visible in this repository.

Concurrency Risk with Replicas >1
This is the highest-risk architectural concern in the service. The scheduler uses optimistic mark-before-send – it sets status = PUBLISHED then publishes to Kafka. With N replicas all running the same cron, the same transaction row can be picked up by two concurrent pods if both execute the select query before either commits the PUBLISHED update.
The findEligibleOutboxRecords JPQL query has no SELECT FOR UPDATE or SKIP LOCKED, meaning two pods can read the same row before either sets it to PUBLISHED. This creates a race window for duplicate Kafka publishing.
The idempotency-key header is the mitigation, but relies entirely on the consumer.
There is no database-level locking, advisory lock, or distributed coordination (e.g. ShedLock, Zookeeper) implemented in this codebase.
