# Deep Technical Research Report: wdp-file-processor

## Executive Summary

wdp-file-processor is a Spring Boot 4.0.3 / Java 17 microservice that acts as the **primary inbound file-ingestion component** for the WDP (Walmart Dispute Platform) chargeback system. It is triggered by AWS SQS S3-event notifications and processes ZIP files from six distinct source acronyms (DWSG, DBLK, DISR, MFAD, DCPO, DNWK), each mapping to a different file format and dispute lifecycle role. For each record it writes to two PostgreSQL tables — wdp.chbk_outbox_row (the chargeback outbox) and 'wdp.file_evidence' (evidence staging) — where a downstream consumer picks up work. It is **not** a Spring Batch application; record iteration is entirely custom. PAN is present in DNWK flat files and is encrypted in-flight via an internal REST encryption service before being written anywhere.

## 1. PRIMARY RESPONSIBILITY

### What It Does
The component listens to an AWS SQS queue for S3 'ObjectCreated' event notifications. When a new inbound ZIP file appears in the S3 bucket, it:
1. Identifies the source acronym from the filename.
2. Opens the ZIP, extracts the index/flat/XML file and any embedded document images.
3. Parses each record and maps it to WDP's internal 'ChbkOutboxRow' (chargeback events) and 'FileEvidence' (evidence attachment metadata) entities.
4. Writes both to PostgreSQL so a separate consumer service can act on them.

### What It Explicitly Does NOT Do
- It does **not** execute the dispute workflow logic (no state machine, no rule engine).
- It does **not** send Kafka events: the downstream consumer reads the outbox table.
- It does **not** perform case look-ups or enrichment lookups against a case system.
- It does **not** generate acknowledgement file content; it only sets 'ack_required' and 'ack_status' flags so a separate process can generate ACK files.
- It does **not** produce outbound network response files.

### Framework
This is a **Spring Boot** application ('FileProcessorApplication'[^1]). There is **no Spring Batch dependency**, no 'Job', 'Step', 'ItemReader', 'ItemWriter', or 'JobLauncher' in the codebase. All record iteration is custom for-loop logic inside service implementations.

### How Files Arrive
All files arrive in an S3 bucket under an '/INBOUND/' path prefix. The inbound check is enforced in 'FileJobUtils.isInBoundMessage(key)' which requires the S3 key to contain '/INBOUND/'[^6].

### Source Identification Logic
Source routing is entirely filename-based:
1. 'FileAcroEnum.fromAcroType(fileName)' — iterates all known prefix patterns and returns the matching enum[^3].
2. If the returned acronym is null or not present in the S3 key path, the file is rejected with error code '912'[^7].
3. For DNWK, sub-routing uses 'NetworkFileEnum' — the filename is checked for substrings 'AMEXHYB', 'AMEXOPTB', 'AMEX', 'MC_REVREJ', 'DISC', 'NYCE', 'DISCHYB' in that priority order[^8]. The matched value is set as 'fileDTO.networkAcroName' and 'NetworkServiceImpl' delegates to the appropriate 'NetworkFileService' bean[^9].

## 3. TRIGGER MECHANISM

**Trigger type**: AWS SQS message listener. One listener, one queue, for all sources.

### SQS Listener Configuration
'SqsMessageListener.receiveMessage(Message message)' is annotated with '@SqsListener'[^10]:
```java
@SqsListener(value = "${cloud.aws.sqs.queue.name}", maxConcurrentMessages = "1", maxMessagesPerPoll = "1")
```
- 'maxConcurrentMessages = "1"' and 'maxMessagesPerPoll = "1"' — **strictly single-threaded, sequential processing**. No parallelism.

### Message Content
The SQS message body is a standard AWS S3 event notification JSON (the same schema Amazon publishes when you configure S3 → SQS event notification). It is deserialized via Jackson into 'Event'[^11] → 'Records' → 'S3'...
- S3 bucket name ('event.records[0].s3.s3Bucket.name')
- S3 object key ('event.records[0].s3.s3Object.key')
- File size in bytes ('event.records[0].s3.s3Object.fileSize' / 'size')

No file type or source identifier is in the message body — both are derived from the S3 key path and filename at processing time[^12].

### Visibility Timeout Extension
Before processing starts, 'sqsAsyncClient.changeMessageVisibility(...)' extends the visibility timeout to '${cloud.aws.sqs.timeout}' seconds[^13]. This prevents the message from reappearing on the queue while a large file is being processed.

### Idempotency Pre-Check
Before invoking processing, 'fileProcessingService.IsProcessedMessage(message.body())' queries 'file_job' for an existing record with the same 'fileName' + 's3Key' where status is not 'PENDING'. If found, the message is immediately deleted from SQS without reprocessing[^14].

## 4. FILE READING AND PARSING

### S3 Access and Streaming
Files are read via AWS SDK v2 'S3Client.getObject(...)', returning a 'ResponseInputStream<GetObjectResponse>'[^15]. The file is **streamed, not loaded fully into memory** as a byte array. For ZIP sources, the stream is immediately wrapped in 'ZipInputStream'.

### S3 Bucket Structure
The bucket names differ per environment. In production[^16]:
- Inbound bucket: 'wdp-files'
- Failed evidence bucket: 'wdp-evidence-failed-files-prod'

The S3 key convention for inbound files is:

<ACRO>/INBOUND/<filename>.zip

During processing, extracted document images are staged to:

<ACRO>/STAGING/<YYYY-MM-DD>/<zipFileWithoutExtension>/<imageName>

After processing completes or errors, the ZIP is archived to:

<ACRO>/ARCHIVED/<MMYYYY>/<filename>.zip

Failed STAGING evidence files are moved to the failed-evidence bucket under:

<sourceEvent>/<YYYY-MM-DD>/<imageName>

### Parsing Per Source

#### DWSG (Walmart Sig Cap) — Fixed-Width, 99 bytes per detail record
- Zip extracted via 'extractIndexFileFromZip()' — '.txt' file — split on '\r?\n|\r' into String[] lines; non-text files uploaded to S3 staging[^17].
- Header: lines[0] starts with "H".
- Trailer: lines[last] starts with "T".
- Detail records: lines starting with "D" — parsed by 'WWChargebackInfo(line)' using fixed substring offsets (total record length = 99 bytes)[^18].

#### MFAD (Discover MAP Notice) — Quote-Delimited Flat File
- Same ZIP extraction.
- No strict header/trailer validation (any non-blank line with alphanumeric chars is processed).
- Parsed by 'DiscoverMapNoticeInfo(line)': splits on '"' (quote), takes 'splitLine[2]', then substring offsets: 'imageName = line.substring(8)', 'caseNumber = line.substring(14, 26)', 'occurrenceNumber = line.substring(26, 28)'[^23].

#### DCPO (Capital One Disputes) — XML (JAXB)
- ZIP extracted via 'extractXmlFromZip()' — '.xml' files — JAXB unmarshalling into 'WpPackage' (generated schema)[^24]; non-XML files → S3 staging.
- 'WpPackage.getMessage()' returns a 'ListMessage'.
- Each 'Message' has a 'messageType' (e.g., 'claim-request', 'outcome-chargeback') and an 'imageList'.
- If filename contains 'BJWC' — 'caseType = "BJPLCC"', 'sourceAcroName = "CAPONE_BJWC"'[^25].

#### DNWK (Network Files) — Fixed-Width COBOL Copybook, 5700 bytes per record
- ZIP containing flat file extracted via 'processZipContainingFlatFile()' — first non-directory ZIP entry read as 'BufferedReader'[^26].
- All records must start with "WDP" — the loop breaks on any line not starting with "WDP".
- Validity: 'NetworkFileSupport.isValidLine(line)' checks that parsed 'eventType' field (bytes 0-25) equals "WDP_INBOUND_EVENT"[^27].
- Full record layout: 'NetworkRecordLayout(line)' — 5700-byte fixed-width COBOL-derived copybook with named field positions[^28].

### Malformed Record Handling
- **Record too short / wrong length**: Sets 'status = ERROR' with appropriate error code. Record is still written to 'chbk_outbox_row' with ERROR status — processing does **not** stop.
- **Image not found in S3 staging map**: Sets 's3Key = "IMAGE NOT FOUND IN S3"', 'status = ERROR'; error code '31' (DWSG/DBLK/MFAD) or '==' (DISR — blank error code, a likely gap).
- **Unparseable ZIP**: Caught at top-level, entire file marked ERROR, archived.
- **JAXBException**: caught per file; file marked ERROR[^29].

### No Maximum File Size or Record Count Limits
No hardcoded maximum file size or record count cap was found in source. The only size validation is a **zero-byte ZIP check** (error code '907')[^30].

## 6. PAN HANDLING

### Which Sources Contain PAN
Only **DNWK (Network File)** sources contain a full account number (PAN). The 'NetworkRecordLayout' field 'acctNum' is extracted from bytes 198-217 of the COBOL record[^28].

Other sources (DWSG, DBLK, DISR, MFAD, DCPO) carry at most the last-4 digits of a card number (DCPO: 'lastFourCardNumber'[^34]). No encryption is performed for those sources.

### Encryption Point
PAN encryption occurs inside 'NetworkFileSupport.mapNetworkFileLayoutToCommonEvent()'[^31], during field mapping, **before** any database write:
```java
if (layout.getAcctNum() != null && layout.getAcctNum().matches("\\d{8}")) {
    originalTransIdentifier.setAccountNumber(encrypt(layout.getAcctNum()));
} else {
    originalTransIdentifier.setAccountNumber(layout.getAcctNum());
}
```

### Encryption Call Flow
1. 'NetworkFileSupport.encrypt(pan)' calls 'restInvoker.encryptField(encryptionUrl, request, idpToken)'.
2. 'EncryptionRequest' contains '{ "pan": "<cleartext>" }'.
3. 'EncryptionResponse' returns '{ "hpan": "<encrypted>" }'.
4. The 'hpan' value is assigned to 'originalTransIdentifier.accountNumber'.

### What Is Written
- The **hpan (encrypted/tokenised PAN)** is serialised into the 'payload' JSON column of 'chbk_outbox_row'.
- **Clear PAN is never written to any store.** If the encryption call fails or returns null, 'accountNumber' is set to 'null'[^31]; an exception causes the record to be written with ERROR status.

### Encryption Failure Handling
'encrypt()' throws a 'RuntimeException' wrapping the original exception if the REST call fails[^31]. This propagates up to the per-record try/catch in 'AmexNetworkFileServiceImpl' (and peers), which saves the record with 'errorStatus = ERROR', 'errorCode = DETAIL_ROW_INVALID_ERROR_CODE ("35")'[^35].

## 7. PROCESSING LOGIC PER FILE TYPE

### Common Pattern (All Sources)
1. SQS listener receives S3 event JSON.
2. 'FileProcessingServiceImpl.processDataFromMessage()' — parses event, extracts S3 key, derives 'sourceAcro' from filename.
3. 'insertFileJobTable()' — inserts 'file_job' row with 'status = PENDING'.
4. Downloads file from S3 as 'ResponseInputStream'.
5. Source-specific method ('processIndexZipFile', 'processXmlZipFile', 'processZipContainingFlatFile') extracts content.
6. Dispatches to 'AcroService' implementation via the 'Map<String, AcroService>' acroServices bean map keyed by service bean name.
7. Service iterates lines/records from 'rowStartNumber' (supports resume after pod restart).
8. Per record: maps to 'RowDetail' — 'commonService.insertEvidenceOutBox()' writes 'chbk_outbox_row' + 'file_evidence'.
9. After all records: 'updateStatusToPendingByFileJobIdAndStatusLoading()' — bulk 'LOADING→PENDING' promotion.
10. 'archiveZipFile()' — moves ZIP from 'INBOUND/' to 'ARCHIVED/<MMYYYY>/'.
11. 'updateFileJobTable()' — sets final status on 'file_job'.

### DWSG (Walmart Sig Cap)
- Processes lines starting with "D".
- Parses fixed-width 'WWChargebackInfo' (99 bytes).
- Case number: bytes 1-13 (12 chars). PIN indicator: bytes 69+.
- 'formatCaseNumber()': if all numeric — truncate to 10 chars (strips occurrence number); if alphanumeric — trim to 12.
- Source system: first char of case number — 'P' = 'PIN'; 'C' or numeric default = 'CORE'.
- 'caseStage': 'caseNumber.substring(10, 12)' if case number ≤ 10 digits.
- Image lookup: key = imageName + '.tiff' in 's3KeyMap'.
- **PIN indicator special branch**: if 'pinIndicator = "y"' → 'status = SUCCESS' (no validation), copies document to '<ACRO>/PINDISPUTE/<date>/<pinNetwork>/<imageName>' via 'pinDocumentS3Transfer()'[^38].
- ACK required if rowCount ≥ 0.

### DBLK (Core Bulk Response)
- Detail lines: start with "2" AND length > 102.
- 'chartsNumber' (case number) = bytes 9-21; 'imageName' = bytes 100+.
- Same 'formatCaseNumber()' / 'getSourceSystem()' logic.
- EventType = 'EVIDENCE_ATTACH'. DocumentType = 'RESPDOC'. SourceSystem = always 'CORE'.
- ACK required if rowCount > 0.

### DISR (Discover MAP Incoming)
- Detail lines start with "2".
- 'ntwkCaseNumber' = bytes 6-16; 'imageName' = bytes 44-94 (50 chars).
- No 'caseNumber' set; 'setNetworkCaseId' set instead.
- CardNetwork = 'DISCOVER'. SourceSystem = 'ISSRDOC'. 'DocumentType = 'CORE'.
- 'caseNumber' field is left **null** (no WDP case ID from this file).
- ACK not required ('getAckRequired(false)').
- No image size > limit check.

### MFAD (Discover MAP Notice)
- All non-blank lines processed (no header/trailer markers checked).
- Line split by '"' — third segment: 'imageName = line.substring(8)', 'caseNumber = line.substring(14, 26)', 'occurrenceNumber = line.substring(26, 28)'.
- 'isValidDocument' flag gates downstream status — if false → 'ERROR', code '35'.
- 'formatCaseNumber()' variant: leading-zero stripping for numeric, space-truncation for alphanumeric.
- DocumentType = 'DSNOTDOC' (dispute notice document). ACK not required.

### DCPO (Capital One Disputes)
This is the most complex source. Each 'Message' in the XML produces **two types of outbox records**:

**a) Chargeback record (CHARGEBACK_PROCESS)**:
- 'caponeChargebackSupport.mapMessageToCommonEvent()' builds 'CommonEvent'.
- 'messageType == "claim-request"' → 'isCreateCase = true' → 'disputeStage = "REQ"', 'sourceSystemCaseId = acquirerClaimId'.
- Otherwise → 'disputeStage = "CH1"'.
- CardNetwork = 'CAPONE'. CaseType = 'BJPLCC' (for BJWC files).
- 'chbkOutboxRowRepository.save(chbkOutboxEntity)' — **direct save without 'insertEvidenceOutBox'**.

**b) Evidence records (EVIDENCE_ATTACH)** — one per 'ImageMeta' in the message:
- 'rowDetail.setStatus(StatusEnum.BLOCKED.name())' — **evidence is always written BLOCKED**.
- 'rowDetail.setRecordDetail(null)' — no record detail in evidence row.
- Parent row number = parent chargeback row's rowNumber.
- 'commonService.insertEvidenceOutBox(rowDetail, s3KeyMap, fileJobId)' writes 'chbk_outbox_row' + 'file_evidence'.

### DNWK (Network Files — AMEX, AMEXHYB, DISC, MC_REVREJ, NYCE, DISCHYB)
All sub-implementations follow identical logic (same code with different service bean names):
- Lines must start with "WDP" (loop breaks on first non-WDP line).
- Validity: 'eventType' field (bytes 0-25) must equal "WDP_INBOUND_EVENT".
- **PAN encryption** called per record.
- 'NetworkFileSupport.mapNetworkFileLayoutToCommonEvent()' → 'CommonEvent'.
- 'mapNetworkFileLayoutToDiscoverPayload()' → 'DiscoverPayload' (Discover-specific fields).
- 'buildDiscoverNotes()' → 'DiscoverNotes' (notes XDISCA, XDISCE, XDISCM, MEMMSG).
- If 'schNetCardNetwork' contains 'DISC' → 'discoverPayload' and 'networkNotes' columns populated; otherwise set to null[^37].
- EventType = 'CHARGEBACK_PROCESS'. No evidence rows created.
- ACK not required for network files.
- 'isCreateCase' is hardcoded 'false' in all network services — **never creates a new case from these files**[^38].

## 9. EVIDENCE HANDLING

### Separation of Chargeback vs. Evidence Data
- For **DWSG, DBLK, DISR, MFAD**: Every record IS an evidence record (eventType = 'EVIDENCE_ATTACH'). There are no chargeback records in these files. A ZIP contains a flat index file listing image names plus the image files themselves.
- For **DCPO (Capone XML)**: Each 'Message' element creates one 'CHARGEBACK_PROCESS' row and N 'EVIDENCE_ATTACH' rows (one per '<Image>' element). The chargeback row is created first; the evidence rows reference it via

Image 10
'parentRowNumber'.
- For **DNWK**: Every record is a 'CHARGEBACK_PROCESS' row. No evidence rows are created.

### Evidence Staging S3 Path

<ACRO>/STAGING/<YYYY-MM-DD>/<zipFileNameWithoutExtension>/<imageName>

E.g.: 'DWSG/STAGING/2024-03-15/AUE2_DWSG_POBCDWLJ_WMSIG20240315.zip_without_ext/C123456789.tiff'

### Evidence Linkage to Chargeback Case
- **DWSG / DBLK**: 'i_case' (WDP case number) is extracted from the index file record. The downstream consumer uses 'i_case' to look up the dispute case.
- **DISR**: 'c_ntwk_case_id' is set from formatted 'caseNumber' in the index. 'i_case' is NOT set (remains null).
- **MFAD**: 'i_case' set from formatted 'caseNumber' from the notice line.
- **DCPO**: Evidence 'i_case' = parent chargeback row's 'iCase' (= 'acquirerClaimId'). Linked via 'parentRowNumber'.

### BLOCKED Status Pattern
For **DCPO only**: all evidence rows are written with 'status = BLOCKED'[^47]. This prevents the downstream consumer from attempting to attach evidence before the chargeback case has been created by the consumer processing the parent 'CHARGEBACK_PROCESS' row.

The question of **which component unblocks evidence** is **not visible in this codebase**. The 'wdp-file-processor' only writes BLOCKED; the consumer (a separate service not in this repo) is responsible for querying and unblocking evidence rows after processing the parent chargeback row.

## 10. ERROR HANDLING AND FAILURE MODES

### Transaction Boundary
'CommonServiceImpl.insertEvidenceOutBox()' is annotated '@Transactional'[^48]. The two writes ('chbkOutboxRowRepository.saveAndFlush()' and 'fileEvidenceRepository.saveAndFlush()') are inside a single transaction. If either fails, both are rolled back.

However, the outer record loop in each 'AcroService' has a try/catch that catches 'Exception' and **logs but does not re-throw** — processing continues to the next record[^49]. **A single record failure does not stop the file.**

### Retry Strategy
- **PAN encryption REST calls**: '@Retryable(retryFor = Exception.class, maxAttempts = 3, backoff = @Backoff(delay = 2000ms))'[^50] — 3 attempts, 2-second fixed backoff. If all fail, exception propagates — record written as ERROR.
- **No retry** on S3 read, S3 write, or database operations.
- **No retry** on IDP token retrieval.

### File-Level Failure
If the entire 'processDataFromMessage()' method throws at the top level, the ZIP is archived and 'file_job' updated to ERROR[^51]. The SQS message is NOT deleted, so it will reappear after the visibility timeout. However, upon reappearance, the idempotency check ('IsProcessedMessage') will detect the 'file_job' record with non-PENDING status and delete the message[^52].

### Dead Letter Mechanism
**No DLQ**, no error table, no S3 error path for file-level failures. The only error mechanisms are:
1. 'file_job.status = ERROR' with 'error_code' and 'error_message'.
2. 'chbk_outbox_row.status = ERROR' per record.
3. 'file_evidence.failed_s3_key' for ERROR-status image files moved to the failed-evidence bucket.

Image 11
- **No retry** on IDP token retrieval.

### Partial File Restart
Resume logic exists in all 'AcroService' implementations: on startup of processing for a 'fileJobId', the service queries 'findTopByFileJobIdOrderByRowNumberDesc(fileJobId)' to find the last successfully inserted row[^53]. If found, processing resumes at 'rowNumber + 1'. **This means processing is idempotent at the record level within a file** — if a pod restarts mid-file, it resumes from where it left off (assuming the 'file_job' record still has 'PENDING' status).

## 11. IDEMPOTENCY

### File-Level Idempotency
In 'IsProcessedMessage()' / 'ProcessDataFromMessage()'[^54]:
```java
fileJobRepository.findByFileNameAndS3KeyAndStatusNot(zipFileName, key, StatusEnum.PENDING.name()).isPresent()
```
If a 'file_job' record exists with the same 'fileName' + 's3Key' AND status ≠ 'PENDING', the SQS message is deleted immediately and the file is NOT reprocessed. This covers duplicate SQS delivery after successful processing.

**Gap**: If the file is currently in status 'PENDING' (e.g., a previous run inserted the file_job row but crashed before processing), the file will be reprocessed from row 0 unless records already exist in 'chbk_outbox_row' (in which case the resume logic skips ahead). If 'chbk_outbox_row' has partial records but 'file_job.status = PENDING', a duplicate of records 1 through last-recorded-row will NOT be inserted (resume skips them), but the 'file_job' row will be inserted as a duplicate — actually no: 'insertFileJobTable' also checks 'findOneByFileNameAndS3Key' and returns the existing ID if found[^55].

**Full duplicate risk per source**: Low for most sources due to the dual guard (file_job unique check + record resume). The main unguarded scenario is if two pods receive the same SQS message simultaneously — both may pass the 'IsProcessedMessage' check before either writes the 'file_job' row. Since 'maxConcurrentMessages = 1' and 'maxMessagesPerPoll = 1', this is mitigated by design but not by a database unique constraint.
