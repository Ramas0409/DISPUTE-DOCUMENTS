## Executive Summary

`gcp-visa-issuer-decline-batch` is a Spring Batch application that runs on a configurable cron schedule to process Visa issuer pre-arbitration decline cases. It reads queued action records from a PostgreSQL database (`nap.action`), enriches each record via REST calls to three internal microservices (Case Management, Visa Adapter, and Case Actions), and writes a new IDCL-coded draft action back to the Case Actions service. The job is a **VISA-only, NAP-platform** workflow specifically targeting `Pre-Arb Declined` dispute states; all non-VISA cases and cases without a valid `disputePreArbResponseId` are silently skipped. No Kafka or file I/O is involved in the current implementation despite the `spring-kafka` dependency being present in the POM.

## 1. RESPONSIBILITY

### What This Job Does

The batch job polls the `nap.action` table in a PostgreSQL database for open Pre-Arbitration (PAB) dispute action records that have been migrated and are of type `OPAB`. For each qualifying record it calls the internal Case Management service to verify the card network is **VISA**, queries the internal Visa Adapter for the Visa HyperSearch case details, and — if the VISA case is in a `Fraud Dispute - Pre-Arb Declined` or `Authorization Dispute - Pre-Arb Declined` state — creates a new `IDCL` (Issuer Decline) action in the Case Actions service to advance the dispute workflow.

### What It Explicitly Does NOT Do

- **Does not process non-VISA cards.** Any case whose `cardNetwork` is not "VISA" (e.g., Mastercard, Amex) is immediately skipped with `skipCase=true`.[^1]
- **Does not handle partial decline or acceptance states.** Only exact matches on "Fraud Dispute - Pre-Arb Declined" and "Authorization Dispute - Pre-Arb Declined" trigger an action; all other `stageStateDesc` values are skipped.[^2]
- **Does not write to any database directly.** The job reads from the DB but all writes are delegated to the Case Actions REST service.[^3]
- **Does not produce or consume Kafka messages.** Despite `spring-kafka` and `aws-msk-iam-auth` being declared in the POM, the dependencies appear vestigial or staged for future use. `KafkaTemplate`, `KafkaListener`, `@SendTo` etc. are absent.[^4]
- **Does not update the `nap.action` table** upon processing. There is no status update (e.g., marking rows as `PROCESSED`) after the action is created downstream.
- **Does not process Mastercard, Amex, or any non-NAP platform cases.**

## 2. TRIGGER AND SCHEDULE

### What Triggers the Job

The job is triggered by a **Spring `@Scheduled` cron expression** on the `BatchScheduler.run()` method.[^5]

### Exact Schedule / Interval

The cron value is **not hard-coded**; it is externalized via the environment variable / property `${scheduler_cron}` bound to `app.scheduler.cron`:
```yaml
# application-prod.yaml (line 16)
app:
  scheduler:
    cron: ${scheduler_cron}
```

The actual value is injected at runtime via Kubernetes secrets (`gcp-visa-issuer-decline-batch-secrets`) or environment-specific override.[^6] The cron expression format is standard Spring cron (`second minute hour dom month dow`).

## 5. ERROR HANDLING

### Retry Strategy

**No explicit retry configuration** (`@Retryable`, `RetryTemplate`, or Spring Batch `.faultTolerant().retry()`) is present in the codebase. The `spring-retry` dependency is declared in the POM[^22] but is unused in the current implementation.

### Per-Record Failure Behavior

Spring Batch is configured with **chunk size = 1** (`app.batchProperties.chunk=1`).[^23] This means each item is individually committed.

In `BatchItemProcessor.process()`:
- Any `Exception` during processing is **caught and swallowed**, returning `null`.[^24]
- Spring Batch treats a `null` return from a processor as "skip this item" (the item is not passed to the writer).
- **One record failure does NOT stop the batch.** The next item continues to be processed.

In `BatchItemWriter.write()`:
- Per-item exceptions are **caught and logged** (`log.error(...)`) within the for-each loop.[^25]
- Processing continues to the next item.
- **A writer-level failure for one case does not stop the batch.**

### Dead Letter / Error Tracking

- **No dead-letter queue.** Failed items are not sent to a DLQ, error table, or retry queue.
- **No database status update on failure.** The `nap.action` row remains `C_ACTION_STA = 'OPEN'` and `C_MIGRATION_STA = 'Y'`, meaning it will be **picked up again on the next scheduled run** (if still within the `pastDays` window).
- Errors are logged via SLF4J/Logback to console and Logstash.

### Alerting / Monitoring Hooks

- **OpenTelemetry:** The Kubernetes deployment manifest injects the OTel Java agent via annotation `instrumentation.opentelemetry.io/inject-java:`

IMAGE 10
- **OpenTelemetry:** The Kubernetes deployment manifest injects the OTel Java agent via annotation `instrumentation.opentelemetry.io/inject-java: opentelemetry-operator-system/default`.[^26] This provides distributed tracing and metrics.
- **Spring Actuator:** Included as a dependency, providing `/actuator/health`, `/actuator/info` etc. on port 8082 for liveness/readiness probes.
- **Logstash ELK:** All log output (including errors) is shipped to the ELK stack for alerting.
- **No custom `JobExecutionListener` or `StepExecutionListener`** is registered; batch job outcome is logged at INFO level by the scheduler only.[^27]

### IDP Token Failure

If the IDP token call fails, a `RuntimeException` is thrown from `IdpTokenServiceImpl`, which propagates to the `StepTokenHolder`, ultimately causing the REST call (and thus the `process()` or `write()` step) to fail. Because exceptions are caught at the processor/writer level, the individual record fails silently; however if the token is unavailable for the entire run, all records will fail silently.

## 6. IDEMPOTENCY

### Mechanism Analysis

**There is no explicit idempotency mechanism** in the application code. Specifically:

1. **No "processed" status flag is set** on `nap.action` rows after successful processing.
2. **Records will be re-queried on every run** as long as `C_ACTION_STA = 'OPEN'` and `C_MIGRATION_STA = 'Y'` and `Z_INSRT > (now - pastDays)`.
3. **Duplicate action creation is therefore possible.** If the job runs twice (e.g., due to a pod restart mid-run, multiple replicas, or overlapping cron windows), the same `nap.action` records will be read again and `addAction` will be called again against the Case Actions service.

**Only partial de-duplication comes from:**
- The downstream Case Actions service (`mdvs-gcp-case-actions-service`) — it may enforce its own duplicate prevention, but this is outside the scope of this codebase.
- Spring Batch's `JobRepository` — the job parameters include a timestamp (`date=yyyyMMdd_HHmmss.SSS`), ensuring each scheduled invocation creates a new `BATCH_JOB_INSTANCE`. This prevents re-running the same logical job instance, but does **not** prevent re-processing the same source records in a new run.[^28]

**Kubernetes replicas:** The `resources.yaml` `spec.replicas` is parameterized (`{{ replicas-gcp-visa-issuer-decline-batch }}`). If more than 1 replica is deployed, multiple instances will fire simultaneously on the same cron schedule, reading the same records and potentially creating duplicate actions. No distributed locking is implemented.

## 1. VOLUME

### Expected Records Per Run

The volume of records processed per run is **not deterministic from source code alone**. All volume-related parameters are externalized:

| Parameter | Property Key | Source |
|-----------|-------------|--------|
| Look-back window | `app.batchProperties.pastDays` | `${past_days}` env var |

- The first query reads all rows from `nap.action` where `Z_INSRT > (now - pastDays)` and the four status filter columns match, sorted by `Z_INSRT` ascending, limited to `numberOfRecords`.[^1]
- The **unit test mock** uses `pastDays = 7` and `numberOfRecords = 5` as example values — this is the only concrete number in the source and is a test fixture, not a production value.[^2]
- There is no maximum total record cap enforced in application code. The reader uses cursor-based pagination; once a page is exhausted, it fetches the next page using `I_ACTION_ID > lastSeenId`, continuing until no more records are returned.[^3]

### Batch (Chunk) Size

The Spring Batch chunk size is **hardcoded to `1`** in every environment configuration YAML.[^4] This means:
- Each `ActionEntity` is read, processed, and written **individually** in its own transaction.
- There is no batching of multiple records into a single commit.
- Each record triggers up to **4 REST API calls**: Case Management GET, Visa Adapter HyperSearch POST, Case Actions GET, Case Actions POST.
