Research Report: gcp-core-hierarchy-authorization-service
Executive Summary
The core-hierarchy-authorization-service is a Spring Boot REST service (Java 17, v1.0.7) within Worldpay's WDP (Worldpay Dispute Platform). Its primary job is JWT-based authorization: it checks whether a caller's JWT token grants them access to a specific merchant, chain, or hierarchy entity by cross-referencing token claims against two backend databases. Secondarily, it acts as a merchant data read API, exposing merchant hierarchy structures, entity-type searches, and product entitlement lookups. The service does not issue tokens, process disputes, or modify any data. All database access is read-only.

1. Primary Responsibility — What It Does and Does Not Do
What It Does
A. JWT Authorization (Core Purpose)
The service validates that a caller's JWT token grants access to a given merchant entity or entity hierarchy. 1 Two authorization flows exist:

Simple authorization (POST /authorize): Given a platform, a case number (or merchantId/level4Entity), the service extracts iqorgid and iqentities claims from the JWT and confirms the requested entity (chain or merchant) is within the caller's organizational scope by querying the Core DB. 2
Entity hierarchy authorization (POST /entity-authorize): Given an explicit hierarchy structure of parent/child entities (chains, super-chains, merchants), the service verifies every entity value in the hierarchy belongs to the caller's scope. 3

Special bypass: if the JWT iss claim contains us_worldpay_fis_int, the caller is treated as an internal firm and authorization is automatically granted without database lookups. 4
B. Merchant Hierarchy Data API
The service provides read-only lookups from the Core DB:

Retrieve the full merchant entity hierarchy for a given Org ID. 5
Retrieve entities filtered by type (MT/CH/SC/ST/PT/DV) scoped to the JWT caller's entity set. 6
Retrieve merchant details for a given merchant ID. 7

C. Product Entitlement Lookup
Returns RDR/Defender product entitlement data (opt-in/opt-out dates, product names, pre-note indicator) for a merchant by chain and/or MID. 8
What It Does NOT Do

Does not issue JWT tokens — it is an OAuth2 Resource Server only. 9
Does not process or manage disputes — it has no dispute creation or mutation logic.
Does not write to any database — all DB access is read-only (JPA queries, no save/delete calls).
Does not serve NAP platform for the simple authorization path — NAP requests are explicitly rejected with a 400 BadRequest. 10
Does not expose Swagger UI in production — the Swagger endpoints are whitelisted only in non-prod environments. 11


2. Inbound Interfaces
Protocol and Security
All inbound communication is HTTP REST secured by OAuth2 JWT Bearer tokens. The service is configured as a Spring Security OAuth2 Resource Server. 9 It validates tokens against a configured list of trusted JWT issuers (jwt.trustedIssuers from application.yaml). 12
All endpoints (except actuator health probes and, in non-prod, Swagger docs) require a valid Bearer token. 11
An optional v-correlation-id request header is accepted for tracing. 13

Research Report: Access Control, /authorize Contract, CRUD vs Auth-Only, and Data Ownership
Executive Summary
core-hierarchy-authorization-service enforces a two-layer access control model: first, every request must carry a valid OAuth2 JWT from a trusted issuer (network layer, enforced by Spring Security); second, for business authorization calls, the service validates the JWT's embedded entity claims (iqorgid/iqentities) against the merchant hierarchy stored in the Core DB2 database. The /authorize endpoint accepts a platform, case number, merchant ID, and/or chain ID; resolves the target merchant through the WDP PostgreSQL case store if a case number is given; then verifies the token's entities match in the Core DB. The service is strictly authorization + read-only — it owns no data, performs no writes, and exposes no CRUD operations. All data it queries belongs to either the Core merchant hierarchy system (DB2) or the WDP dispute platform (PostgreSQL).

1. How Access Control Is Enforced
Access control is applied in three sequential layers. A request must pass all layers to succeed.
Layer 1 — Transport-Level JWT Validation (Spring Security)
All endpoints except health probes (/actuator/health, /livez, /readyz) and, in non-prod, Swagger docs require a valid Authorization: Bearer <token> header. 1
The service is configured as an OAuth2 Resource Server using Spring Security's multi-issuer JWT support: 2
javaJwtIssuerAuthenticationManagerResolver authenticationManagerResolver =
    JwtIssuerAuthenticationManagerResolver.fromTrustedIssuers(trustedIssuers);
http.authorizeHttpRequests(authorize ->
    authorize.requestMatchers(authWhiteList).permitAll().anyRequest().authenticated())
    .oauth2ResourceServer((oauth2) ->
        oauth2.authenticationManagerResolver(authenticationManagerResolver));
The trusted issuers list comes from the jwt.trustedIssuers config property (${jwt_trusted_issuer_urls}). 3 If the token is missing, expired, or signed by an untrusted issuer, Spring returns 401 Unauthorized before the application code is ever reached.
Layer 2 — Internal Firm Bypass
Within the business authorization logic, before any database query, the service checks the JWT iss (issuer) claim for the value us_worldpay_fis_int: 4
javaif (StringUtils.containsAnyIgnoreCase(
    (String) jwt.getClaims().get("iss"),
    ApplicationConstants.INTERNAL_FIRM)) { // "us_worldpay_fis_int"
    log.info("Authorizing token as internal firm");
    return true;
}
If matched, authorization is unconditionally granted. This is the "internal service account" bypass, allowing Worldpay-internal systems to call the service without entity-level scoping.
Layer 3 — Entity-Scope Check Against Core DB
For all non-internal callers, the service reads two JWT claims:
ClaimMeaningiqorgidThe caller's org ID (integer). Used when iqentities is absent.iqentitiesSemicolon/comma-delimited list of entity IDs the caller has access to. Takes priority over iqorgid.
5
The service then runs one of two Core DB queries to confirm the requested entity falls within the caller's scope:
When iqentities is present: 6
sqlSELECT TRIM(en.I_LCL_ENTY)
FROM MD.TMD_ENTY en, MD.TMD_ENTY_REL re
WHERE re.I_CHILD_ENTY = en.I_ENTY
AND re.C_CHILD_TYPE = :entityType        -- 'CH' or 'MT'
AND re.I_PARNT_ENTY IN :parentEntities   -- the iqentities list
AND en.I_LCL_ENTY IN :localEntities      -- the requested entity value(s)
When only iqorgid is present: 7
sqlSELECT TRIM(en.I_LCL_ENTY)
FROM MD.TMD_ENTY en, MD.TMD_ENTY_REL re
WHERE re.I_CHILD_ENTY = en.I_ENTY
AND re.C_CHILD_TYPE = :entityType
AND re.I_PARNT_ENTY IN (
    SELECT og.I_ENTY FROM DC.TDC_VIQ_ORG_ENTY og WHERE og.I_ORG_ID = :orgId
)
AND en.I_LCL_ENTY IN :localEntities
The logic is: "Does the Core DB's entity relationship tree confirm that the requested entity (merchant/chain) is a descendant of the entities the caller's token is scoped to?"
If the DB returns no matching rows, or if the returned set does not contain the requested entity value, a 403 FORBIDDEN is returned. 8
Entity Type Priority Logic
Within authorizeToken, the service first attempts to validate via the chain (level4Entity / CH type). Only if that is unavailable does it fall back to the merchant (levelEntity / MT type): 9
javaif (StringUtils.isNotBlank(level4Entity)) {
    // Try chain first (ENTITY_CHAIN_TYPE = "CH")
    isAuthorized = getFromCoreEntityRelations(..., "CH", ...);
} else {
    throw new Exception(); // triggers catch block — fall through to merchant
}
// catch block:
if (StringUtils.isNotBlank(levelEntity)) {
    // Fall back to merchant (ENTITY_MERCHANT_TYPE = "MT")
    isAuthorized = getFromCoreEntityRelations(..., "MT", ...);
}

2. The /authorize Endpoint — Complete Contract
Request
Method: POST
Path: /authorize
Authentication: OAuth2 Bearer JWT (required)
Content-Type: application/json
Request Body (AuthorizationRequest): 10
json{
  "platform":     "PIN",          // Required. One of: CORE, NAP, VAP, LATAM, PIN
  "caseNumber":   "P123456789",   // Optional. Dispute case number.
  "merchantId":   "123456789",    // Optional. Level-1 entity / merchant ID.
  "level4Entity": "070111"        // Optional. Chain ID (level-4 entity).

Validation Rules: 11

platform is @NotBlank — must be present and non-empty.
platform must be one of the SourceSystemName enum values (CORE, NAP, VAP, LATAM, PIN). Invalid → 400.
If platform != NAP: at least one of caseNumber, merchantId, or level4Entity must be provided. All three missing → 400.
platform == NAP → always 400 (NAP is not supported for this flow).

Internal Processing Steps

Map to internal request: AuthorizationMapper.formatAuthorizationRequest() copies the request to an EntityAuthorizationRequest object (same fields plus entityHierarchy which is null here). 12
Check JWT existence: If jwt == null or jwt.getClaims() == null → 400 Bad Request. 13
Internal bypass check: If iss contains us_worldpay_fis_int → return 200 OK immediately.
Resolve case if caseNumber provided: Query USCaseRepository.findByCaseNumber(caseNumber) against WDP.CASE (PostgreSQL). Extracts C_LEVEL1_ENTITY and C_LEVEL4_ENTITY from the case row. 14

Case not found → 403 Forbidden ("Case number not found").
Case found but both entities are blank → 403 Forbidden.


Determine entity to validate: Uses level4Entity from case or request (preference: level4 = chain). Falls back to levelEntity/merchantId if level4 is blank.
Core DB entity relation query: See SQL above. Validates entity is in caller's scope using iqentities (list) or iqorgid (single org).
Authorization decision: If DB returns the requested entity value in its result set → authorized. If empty or not containing the value → 403.

3. Authorization Only vs. CRUD — Is Any Data Modified?
The service is authorization and read-only queries only. It performs zero writes.
Evidence

No save, delete, or update calls anywhere in the codebase. All repository methods used are query-only:

USCaseRepository.findByCaseNumber() — SELECT 16
OrgEntityRepository.getLocalEntitiesByOrgId() / getLocalEntitiesByParentEntity() — SELECT 6
MerchantRepository.findByMerchantId() — SELECT 17
All MerchantDaoImpl methods — native SELECT queries with with ur (Uncommitted Read, DB2 read-only hint) 18
ChainRepository.findTopByChain() — SELECT 19
MerchantWomplyRepository.findFirstByProductAndMid() — SELECT 20


DB2 dialect configured with spring.jpa.hibernate.ddl-auto: false — Hibernate will not attempt to create or alter any schema. 21
No controller methods use HTTP PUT, PATCH, or DELETE — the three controllers expose only POST (for queries that require body) and GET. 22
No JPA entity has @GeneratedValue strategy used for inserts (the USCaseEntity has one but it is never invoked — the entity is only fetched, never constructed for saving). 23

What the Service Does Do (Read-Only)

Reads merchant hierarchy data from the Core DB2 (MD.*, DC.*, BC.* schemas)
Reads dispute case data from the WDP PostgreSQL DB (WDP.CASE table)
Returns JSON responses derived from those reads


4. Data Ownership and Persistence Scope
What This Service Owns: Nothing
The service owns no database tables and persists no data. It is a pure read-through service.
Data It Reads From (Not Owns)
Source 1: WDP PostgreSQL Database — Dispute Case Data

Research Report: Pod Scaling, Memory Configuration, and Architectural Risks
Executive Summary
The service's replica count is a deploy-time template variable ({{ replicas-core-hierarchy-authorization-service }}); the exact value is not stored in this repository and varies by environment. Memory is explicitly bounded at 2048Mi limit / 1024Mi request, but no CPU limits or requests are configured, and there is no Horizontal Pod Autoscaler. The codebase contains several architectural concerns: a label mismatch that silently disables the topology spread constraint, no connection pool tuning across two databases, no circuit breaker on the external Core DB2 dependency, use of exception-throwing for control flow in the authorization path, dead injected beans, and no caching on hot authorization paths.

1. Pod Count and Scaling Configuration
Replica Count
The number of pods is not hardcoded in this repository. The resources.yaml Kubernetes Deployment spec uses a placeholder:
yamlspec:
  replicas: {{ replicas-core-hierarchy-authorization-service }}
1
This value is resolved at deploy time by XL Deploy (the CI/CD tool referenced in deployit-manifest.xml 2 and Jenkinsfile 3). The actual replica count per environment is stored in XL Deploy configuration external to the repo. This codebase alone cannot tell you how many pods run in production.
Rolling Update Strategy
yamlstrategy:
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
  type: RollingUpdate
1

maxUnavailable: 0 — No pod is taken down before a new one is ready. Zero-downtime deployments.
maxSurge: 1 — At most one extra pod is created above the desired count during a rollout.
minReadySeconds: 30 — A new pod must be running without crashing for 30 seconds before being considered available. 1

The JVM process (Spring Boot fat JAR) has no JAVA_OPTS or -Xmx/-Xms flags set anywhere in the deployment manifest. 2. Architectural Risks and Concerns
Risk 1 — Topology Spread Constraint Label Mismatch (Bug)
Severity: Medium — The topology spread constraint in resources.yaml uses a labelSelector with a different label value than the pods it is supposed to spread:
yaml# Deployment and pod labels use:
app: core-hierarchy-authorization-service${BRANCH_NAME_PLACEHOLDER}

# Topology spread constraint labelSelector uses:
app: gcp-core-hierarchy-authorization-service${BRANCH_NAME_PLACEHOLDER}
#    ^^^^ extra "gcp-" prefix
1
The gcp- prefix appears in the topology spread constraint but not in the deployment's matchLabels or pod template labels. As a result, the spread constraint's labelSelector will never match any pods, rendering the constraint inoperative. Pods can be scheduled on the same Kubernetes node without any enforcement of spread. In a cluster with limited nodes, this could result in all replicas landing on the same node — a single-node failure would take down all instances simultaneously.
Risk 2 — No CPU Requests or Limits
Severity: Medium — The pod spec sets only memory requests/limits. No CPU request or limit is defined: 1
yamlresources:
  limits:
    memory: 2048Mi
  requests:
    memory: 1024Mi
    # CPU: not set
Without CPU requests, the Kubernetes scheduler has no signal for bin-packing CPU workloads. Without CPU limits, a pod can consume all CPU on its node, causing noisy-neighbor effects on other workloads. Additionally, the OpenTelemetry Java agent adds non-trivial CPU overhead that is entirely unconstrained.
Risk 3 — No Connection Pool Configuration (Both Datasources)
Severity: Medium — Neither the application.yaml nor any of the environment-specific YAML files configure HikariCP pool settings for either datasource. 5 The service connects to two separate databases (PostgreSQL for WDP cases, DB2 for Core hierarchy) using default Hikari parameters:

Default maximum pool size: 10 connections per datasource
Default connection timeout: 30 seconds
No minimumIdle, maxLifetime, or connectionTimeout tuning

Under load, if authorization requests arrive faster than 10 concurrent DB queries can complete, requests will queue waiting for a connection. This is compounded by the fact that a single /authorize request may touch both databases (case lookup on PostgreSQL + entity check on DB2), effectively consuming connections from two pools simultaneously.
Risk 4 — No Circuit Breaker on Core DB2 Dependency
Severity: High — Every business authorization call depends on the Core DB2 being available and responsive. There is no circuit breaker (Resilience4j, Hystrix, or similar), no retry, and no fallback path. 10 If the Core DB2 becomes slow or unreachable:

All connections in the DB2 Hikari pool will be held waiting.
New requests will queue at the pool, timing out after 30 seconds.
The service will return 500 Internal Server Error (or eventually 503) to all callers until DB2 recovers.
The liveness probe (which checks /livez — a Spring Boot actuator endpoint, not a DB connectivity check) will continue to return 200, so the pod will not be restarted.

There is no graceful degradation.
Risk 5 — Exception Used for Flow Control in Authorization Path
Severity: Low-Medium — In AuthorizationServiceImpl.authorizeToken(), a bare throw new Exception() is used to trigger the fallback from chain-based to merchant-based entity lookup: 11
javatry {
    if (StringUtils.isNotBlank(level4Entity)) {
        // ...
        isAuthorized = getFromCoreEntityRelations(..., "CH", ...);
    } else {
        throw new Exception(); // signals "no level4Entity, fall through"
    }
} catch (Exception e) {
    if (StringUtils.isNotBlank(levelEntity)) {
        isAuthorized = getFromCoreEntityRelations(..., "MT", ...);
    }
}
This pattern:

Swallows any genuine exception from the getFromCoreEntityRelations call on the chain path (e.g., a DB timeout would silently fall through to trying the merchant path rather than propagating the error)
Makes it impossible to distinguish between "level4Entity was blank" and "the DB call threw an error"
Reduces stack trace visibility in logs

Risk 6 — Dead Beans / Dead Code
Severity: Low
RestTemplate bean declared but never used:
CommonConfig defines a RestTemplate Spring bean: 12
java@Bean
public RestTemplate getRestTemplate() {
    return new RestTemplate();
}
No service class, repository, or controller injects or uses RestTemplate anywhere in the codebase. This is dead infrastructure code, likely left over from a previous integration that was removed or never completed.
ProductEntitlementRepository injected but never called:
ProductEntitlementServiceImpl autowires ProductEntitlementRepository, 13 but never calls any of its methods. The actual product entitlement data is fetched via MerchantWomplyRepository and ChainRepository. The ProductEntitlementServiceImpl.findFirstByChainAndProduct() method is a dead path.
Risk 7 — No Caching on Hot Authorization Paths
Severity: Medium — Every call to /authorize or /entity-authorize results in at least one (usually two) synchronous database queries:

USCaseRepository.findByCaseNumber() — PostgreSQL query (if case number provided)
OrgEntityRepository.getLocalEntitiesByParentEntity() or getLocalEntitiesByOrgId() — DB2 native SQL query

For WDP dispute UIs that call /authorize on every page load or action, this means N uncached DB round-trips per user action. There is no @Cacheable, no Redis, and no in-process cache (Caffeine, etc.) anywhere in the codebase. 14 High-frequency callers will generate proportionally high DB load.
Risk 8 — Logstash TCP Appender Without Fallback
Severity: Low — The logback-spring.xml configures a LogstashTcpSocketAppender that writes logs directly to a TCP Logstash endpoint: 15
xml<appender name="awselk" class="net.logstash.logback.appender.LogstashTcpSocketAppender">
    <destination>${LOGSTASH_SERVER_HOST}:${LOGSTASH_SERVER_PORT}</destination>
    <keepAliveDuration>5 minutes</keepAliveDuration>
</appender>
All loggers — including the root logger — send to both console and this TCP appender. If the Logstash endpoint is unavailable:

Log messages will back up in the TCP socket buffer
Application threads can block waiting for the socket buffer to drain
There is no ringBuffer size limit or neverBlock configuration to prevent this
There is no fallback-only-to-console configuration

Risk 9 — Snyk Security Scanning Not Enforced
Severity: Low — The Jenkinsfile explicitly sets snykFailOnIssues: 'false': 3
groovysnykFailOnIssues: 'false',
snykSeverity: 'low',
This means Snyk will scan for vulnerabilities down to low severity but will never fail the build regardless of what it finds. The dependency scan result is informational only and does not gate deployment.
Risk 10 — MerchantEntityDetailsHierarchy Is an @Entity Without @Table
Severity: Low — MerchantEntityDetailsHierarchy is annotated @Entity but has no @Table annotation: 16
java@Entity
public class MerchantEntityDetailsHierarchy implements Serializable { ... }
It is used purely as a result mapping target for native SQL queries in MerchantDaoImpl. Without @Table, Hibernate derives the table name from the class name (merchant_entity_details_hierarchy), which does not exist in the DB2 schema. This works in practice only because Hibernate DDL auto-creation is set to false and the entity is never queried through JPQL — but it is fragile. If ddl-auto were ever changed, Hibernate would attempt to create/validate a table that doesn't exist, causing startup failure.
