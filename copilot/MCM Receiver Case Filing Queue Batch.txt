Executive Summary
This is a continuously-running Spring Boot Deployment (not a CronJob) that contains a Spring Batch job triggered on a cron schedule. It polls the MasterCard Connect Merchant (MCM) ReceiverCaseFiling queue for pre-arbitration (Pre-Arb) and arbitration (Arb) dispute events filed by issuers against acquirers/merchants. For each qualifying event it fetches full claim details and settlement transaction data from MCM, optionally encrypts the cardholder PAN, constructs a CommonEvent payload, and inserts a row into the wdp.chbk_outbox_row PostgreSQL table with status PENDING. A separate downstream process (not this batch) consumes the outbox and publishes to Kafka. It handles five distinct dispute lifecycle stages: incoming Pre-Arb, incoming Arb, Pre-Arb/Arb Withdraw, Arb Ruling (issuer favor), and Arb Ruling (merchant favor) — stages that are structurally later in the dispute lifecycle than a First Chargeback.

1. Primary Responsibility
What it does
This batch job polls the MasterCard ReceiverCaseFiling queue via the MCM v6 API to collect dispute cases in which an issuer has filed a Pre-Arbitration or Arbitration case against a Worldpay-side merchant. For each qualifying claim it:

Fetches the full claim detail from MCM
Determines the dispute stage (PAB/ARB/PRA/AII/AIM) and the issuer action
Looks up whether a prior chbk_outbox_row record exists for that claim
Inserts or conditionally inserts a new chbk_outbox_row record with status PENDING (or SKIPPED) and a JSON CommonEvent payload

What it explicitly does NOT do

It does not publish to Kafka — it writes an outbox row and leaves Kafka publishing to a separate component
It does not handle First Chargeback events (those are handled by wdp-mcm-receiver-case-filing-queue-batch's sibling, the First Chargeback batch)
It does not process Compliance (caseType 3 or 4) — those are explicitly skipped1
It does not call any case-lookup or case-creation REST APIs (the caseLookupUrl is configured but never called at runtime — see §13)
It does not write to S3, emit metrics to any external system, or publish any files

The job is not auto-started on application startup (spring.batch.job.enabled: false3). It is exclusively triggered by the @Scheduled cron in BatchScheduler4. Each scheduled execution creates a new JobParameters keyed with a timestamp string (yyyyMMdd_HHmmss.SSS), which ensures Spring Batch treats each trigger as a distinct job instance4.

Difference from First Chargeback Batch
The First Chargeback batch processes caseType events at the initial chargeback stage. This batch only processes caseType "1" (Pre-Arbitration) and caseType "2" (Arbitration)1 — stages that arise after the merchant has already responded to or ignored a first chargeback. The lifecycle stages handled exclusively by this job are:
Stage CodeDescriptionPABIncoming Pre-Arbitration filed by issuerARBIncoming Arbitration or Escalation to ArbPRAWithdrawal (of either Pre-Arb or Arb)AIIArbitration ruling: issuer favor (FAVOR_SENDER)AIMArbitration ruling: merchant favor (FAVOR_RECEIVER)
The issuerResponseType values generated (IncomingPreArb, IncomingArb, ArbRuling) have no equivalent in the First Chargeback pipeline.

All three MCM calls use DataPowerRestInvoker5. The auth header is:

The value is injected from env var ${vantive_license} (K8s secret wdp-mcm-receiver-case-filing-queue-batch-secrets). This is not an OAuth/Bearer token — it is a raw Vantive/MasterCard license key passed directly. A v-correlation-id header with a random UUID is also added per request.

6. PAN Handling
Is PAN present in MasterCard responses? Yes — PAN appears in two MasterCard response fields:

ClaimDetail.primaryAccountNum — from the queue claim detail call11
SettledTransactionDetail.mastercardMappingServiceAccountNumber — from the settled transaction call; preferred over primaryAccountNum14

PAN pipeline (new claims only):

Retrieve: mastercardMappingServiceAccountNumber (or primaryAccountNum fallback)
Trim: if length > 16, take last 16 digits
Extract last 4 digits (accountLastFourDigit) — stored in CommonEvent
Check: if PAN contains "*" — already masked, skip encryption
If clear PAN:
a. POST IDP service → get Bearer token
b. POST /encryption/v1/pan/encrypt { "pan": "<clear_pan>" }
c. Response: { "hpan": "<encrypted_pan>" }
d. Use hpan as accountNumber
accountNumber (encrypted HPAN or masked value) stored in:

CommonEvent.originalTransIdentifier.accountNumber
Serialized into chbk_outbox_row.payload (JSON)



Is clear PAN ever written? No. Clear PAN is never written to any persistent store. It exists only transiently in memory during processing. If encryption fails (exception or blank hpan), a ServiceException is thrown20 and the item processing fails (exception caught at the top level, returns empty ProcessedItem).
For update claims (isNewClaim=false): No PAN lookup, no encryption, no account number fields populated — accountNumber = null, accountLastFourDigit = null15.

8. Error Handling and Failure Modes
Chunk Size and Transaction Boundary

Chunk size = 13 — each record is its own Spring Batch chunk transaction
Transaction manager: wdpTransactionManager (JPA, same PostgreSQL datasource)22
Each chunk commit is independent; one record failure does not roll back other records

Retry Strategy
InvokerRetry countDelayBackoffOn exceptionDataPowerRestInvoker.get()32000msFixedRestTemplateCustomException, ExceptionDataPowerRestInvoker.post()32000msFixedRestTemplateCustomException, ExceptionDataPowerRestInvoker.put()32000msFixedRestTemplateCustomException, ExceptionRestInvoker.post()32000msFixedRestTemplateCustomException, ExceptionIDP token fetchNoneFixedExceptionDB operationsNone———
RETRY_COUNT = "3" and RETRY_DELAY = "2000" are Spring EL expressions used in maxAttemptsExpression and backoffDelayExpression5. There is no exponential backoff — delay is fixed at 2 seconds.
Does one record failure stop the batch? No. Exceptions in BatchItemProcessor.process() are caught at the try/catch block level12; on exception the processor returns an empty ProcessedItem.builder().build() (null entity). The batch continues processing subsequent records.
Similarly, exceptions in BatchItemWriter.write() are caught per-item and logged without re-throwing23.
Dead Letter Mechanism
None. There is no dead letter queue, no error table, and no DLQ. Failed items are silently logged and skipped. There is no record of which claims failed processing in any persistent store other than application logs.
Job-Level Failure and Restart

If BatchScheduler.run() catches an unchecked exception from jobLauncher.run(), it logs and returns BatchStatus.FAILED4
Spring Batch records the failure in the metadata tables (job execution status)
The next scheduled cron execution starts a new job instance (new timestamp parameter) — there is no Spring Batch restart from checkpoint because each run is a new instance
Reader loads all claims fresh on each @BeforeStep execution


9. Idempotency
Mechanism
The batch implements a soft idempotency check based on (networkCaseId, networkPhaseId):

By networkCaseId only: findChbkByNetworkCaseId(standardClaimId) — checks if any outbox row exists for the claim (not phase-specific)24
By networkCaseId + networkPhaseId + status NOT SKIPPED: findByNetworkCaseIdAndNetworkPhaseIdAndStatusNot(...) — checks if a non-SKIPPED row already exists for the exact (networkCaseId, networkPhaseId) combination25

If a non-SKIPPED row exists for the exact (networkCaseId, networkPhaseId) combination, the new item is written with status SKIPPED rather than PENDING. This prevents the downstream Kafka publisher from processing it twice.
Duplicate Risk
Partial duplicate risk exists in two scenarios:

Same claim, different phase: If the same networkCaseId has two different caseFilingDetails.caseId values (e.g., Pre-Arb phase then Arb phase), the first findChbkByNetworkCaseId returns non-empty, triggering the "update" path. The update path then does a finer check by (networkCaseId, networkPhaseId). This is handled correctly.
Concurrent runs: If two instances of the batch run simultaneously (replicas > 1), both could read the same QueueClaim from MCM, both could find no existing outbox row, and both would insert a PENDING row. There is no distributed lock and no optimistic locking on the insert. Duplicate PENDING rows could be produced.
No restart checkpoint: On restart (new job instance), the reader re-fetches all claims from MCM. Claims already in PENDING status from a previous partial run would be found in the outbox lookup and marked SKIPPED (unless their networkPhaseId doesn't match, in which case a new PENDING would be created).
