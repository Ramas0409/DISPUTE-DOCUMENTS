## Executive Summary

`wdp-evidence-ack-scheduler` is a Spring Boot long-running process deployed as a Kubernetes **Deployment** that periodically polls a shared PostgreSQL database for inbound file-processing jobs that have completed (successfully or with errors) and have not yet had an acknowledgement file generated. For each qualifying job it generates a format-specific ACK file and uploads it to AWS S3. Three merchant/source types are supported: **Walmart SigCap** (fixed-width `.txt`), **Meijer / Core Bulk Response** (fixed-width `.txt`), and **CapitalOne** (XML-in-ZIP). No SFTP, API, or messaging layer is involved. The application is version `1.0.3` and the CapitalOne XSD schema is stamped "Version 1.0.0 â€“ 05/02/2026", indicating this component is recently released.

## 1. Primary Responsibility

### What it does

`wdp-evidence-ack-scheduler` is a **scheduled batch processor** that generates outbound acknowledgement files for three merchant sources and uploads them to AWS S3. On every scheduled run it queries the `wdp.file_job` table for jobs that have been processed (status `COMPLETED` or `ERROR`), have `ack_required = true`, and have `ack_status = PENDING`. For each match it reads per-row detail records from `wdp.chbk_outbox_row`, formats them into the merchant-specific file layout, uploads the result to S3, and updates the `file_job` row to `ack_status = COMPLETED`.[^1][^2]

### What it explicitly does NOT do

- It does **not** receive or parse inbound files. Inbound processing is handled by a separate upstream service that populates `file_job` and `chbk_outbox_row`.
- It does **not** deliver ACK files via SFTP, MQ, or API â€” only S3.
- It does **not** process every `file_job` row indiscriminately; `ack_required = false` rows are silently skipped.[^3]
- It does **not** handle Meijer directly by name â€” the source code identifier is `CORE_BULK_RESP`, but the file-name pattern `DBLK_MEJR_CHRGRESP_CONF_` confirms this is the Meijer chargeback bulk-response channel.[^4]

### Which Merchants/sources receive ACKs

| Source constant | Value in DB | Merchant / Program | File format |
|---|---|---|---|
| `SOURCE_WALMARTSIGCAP` | `WALMART_SIG_CAP` | Walmart Signature Capture | Fixed-width `.txt` |
| `SOURCE_COREBULKRESP` | `CORE_BULK_RESP` | Meijer (Core Bulk Response) | Fixed-width `.txt` |
| `SOURCE_CMRTR` | `CAPONE_CMRTR` | CapitalOne Commercial (CMRTR) | XML-in-ZIP |
| `SOURCE_BJWC` | `CAPONE_BJWC` | CapitalOne BJWC | XML-in-ZIP |

Any `file_job.source` value not matching these four constants is logged as "Unidentified source" and skipped.[^5]

## 2. Trigger and Schedule

### Trigger mechanism

The trigger is a Spring `@Scheduled` cron job. The `ScheduledTasks` class is annotated with `@Scheduled(cron = "${app.scheduler.cron}")`. The cron expression is not hard-coded in any deployed profile â€” it is injected at runtime from the environment variable `cron_scheduler` via a Kubernetes Secret reference (`wdp-evidence-ack-scheduler-secrets`).[^6][^7]

The only hard-coded cron expression appears in `application-local.yml`:

app.scheduler.cron: 0 */1 * * * *    # every minute (local/dev only)

Image 3
[^8]

In all deployed environments (dev, stg, uat, cert, prod) the cron value is `${cron_scheduler}` â€” the **actual production schedule is not visible in source code** and must be looked up in the Kubernetes Secret or deployment tooling.[^9]

### Is this a Kubernetes Deployment or CronJob?

**Kubernetes Deployment.** `resources.yaml` declares `kind: Deployment`, not `kind: CronJob`. The scheduling is handled inside the JVM by Spring's `@EnableScheduling` mechanism â€” the pod runs continuously and the cron fires internally.[^10][^11]

## 3. What Triggers an ACK File

### Trigger condition

ACK generation is **schedule-driven**, not event-driven. On each cron fire the processor unconditionally queries the database. If no eligible `file_job` rows exist, it logs "No pending fileJobs found for processing" and exits cleanly.[^2]

### Which inbound files need an ACK

The query in `FileJobRepository` is:
```sql
SELECT f FROM FileJob f
WHERE f.status IN :status        -- ('COMPLETED', 'ERROR')
  AND f.ackStatus = :ackStatus   -- 'PENDING'
  AND f.ackRequired = true
ORDER BY f.id ASC
```

[^3]

This means:
- **Per-inbound-file tracking** â€” each `file_job` row represents one inbound file.
- `ack_required = true` must be set on the record (by the upstream inbound processor) before this scheduler will touch it.
- `ack_status = PENDING` is the "not yet acknowledged" sentinel. Once this scheduler succeeds it flips it to `COMPLETED`.[^12]

The **`wdp.file_job`** table is the ACK-tracking table. Key columns relevant to this scheduler:

## 4. ACK File Generation

### 4a. Walmart SigCap (`WALMART_SIG_CAP`)

**File format:** Fixed-width text (`.txt`)
**Encoding:** UTF-8

**Structure:**
H<yyyyMMdd>\n
D<record_data padded to 78 chars><2-char return code>\n
...repeat for each row...
T<zero-padded total_evidences, 9 digits>

| Section | Indicator | Length | Content |
|---|---|---|---|
| Header | `H` | 9 chars | `H` + date `yyyyMMdd` |
| Detail | (raw) | 80 chars | 78-char padded `recordData` + 2-char return code |
| Trailer | `T` | 10 chars | `T` + zero-padded `totalEvidences` (9 digits) |

**Return code mapping (detail records):**

| Return Code | Meaning | Trigger condition |
|---|---|---|
| `00` | Successfully Processed Record | `row.status = SUCCESS` |
| `30` | Internal Error | Default / unrecognised error code |
| `31` | Document not found | `errorCode = "31"` |
| `32` | Charts(Reference) number not found | `errorCode = "32"` or `errorMessage` contains `CASE_NOT_FOUND` |
| `33` | Server Error | `errorCode = "33"` |
| `34` | Image File exceeds limit of 3 MB | `errorCode = "34"` or `errorMessage` contains `Image Size` or `DOC_SIZE_LIMIT_EXCEEDED` |
| `35` | Invalid Index File Detail Record | `errorCode = "35"` |

7. Error Handling
Scheduler-level error handling
AckFileProcessorServiceImpl.performTask() wraps the entire loop in a try/catch (Exception e). On exception it logs "Unable to proceed" with no stack trace, no re-throw, and no alerting. The scheduler will simply try again on the next cron fire.[^2]
Critical gap: Exceptions inside the per-fileJob loop are also silently swallowed at the individual service level. There is no propagation of per-job errors to any dead-letter mechanism.
Per-service error handling
Each of the three generateAck() implementations has its own try/catch (Exception e) that logs a warning message and returns without updating ack_status. The file_job row remains in ack_status = PENDING, so it will be picked up again on the next run.[^38][^39][^40]
ACK generation failure
If file content generation returns null (no detail records, missing evidence count, null batch header), the service skips the job â€” logs an informational message and returns. ack_status remains PENDING. The job will be re-evaluated on the next cron fire.[^19][^41]
ACK delivery failure
If uploadAckFileToS3 returns false, the updateAckStatus() call is not made. ack_status stays PENDING. No retry is attempted within the current cron execution.[^35]
Retry strategy
De facto retry via repeated polling. There is no explicit retry library (e.g., Spring Retry, Resilience4j). The retry mechanism is entirely implicit: since failed jobs remain ack_status = PENDING, they will be picked up again on the next scheduled run. The maximum number of retries is unlimited and bounded only by the cron schedule.
Risk: If a job consistently fails (e.g., malformed record_detail data), it will be re-attempted on every cron fire indefinitely. There is no poison-pill detection or maximum-retry counter.
Dead letter mechanism
Not implemented. No DLQ, no error queue, no dedicated error table. Failures are log-only.

8. Idempotency
ACK generation level
Partially idempotent. If the scheduler runs twice (e.g., two pods due to a deployment overlap), the following protections exist:

S3 pre-check: uploadAckFileToS3 calls headObject before putObject. If the file already exists at the exact S3 key, the upload is skipped and ack_status is not updated.[^35]
ack_status guard: The DB query only retrieves rows with ack_status = PENDING. Once a job is successfully processed and ack_status is set to COMPLETED, it is never re-selected.[^3]

Race condition risk
The UniqueTimestampGenerator uses a JVM-level synchronized block with an AtomicReference<LocalDateTime>. This ensures unique timestamps within a single JVM, but provides no cross-JVM uniqueness guarantee. If two pods run simultaneously (possible during a rolling update), they could generate ACK files for the same file_job row with different timestamps, leading to two S3 objects and only one ack_status update (the first one to complete would claim COMPLETED; the second would also try to upload a different filename and update status again).[^42]
