Executive Summary
wdp-mcm-first-chargeback-queue-batch is a Spring Boot / Spring Batch application that runs continuously inside a Kubernetes Deployment. Every cron tick it polls a MasterCard Dispute Management (MCM) queue for unworked first chargeback claims via a DataPower proxy, enriches each claim with settled-transaction data, encrypts the PAN through a WDP internal encryption service, and writes a ready-to-publish row into the wdp.chbk_outbox_row PostgreSQL outbox table. It never directly publishes to Kafka — that responsibility belongs to a downstream outbox publisher. The job acknowledges each processed item back to the MCM queue via a PUT call, so it is not re-queued on the next run.

1. PRIMARY RESPONSIBILITY
What it does
The batch pulls open, unworked first chargeback items from the MasterCard Dispute queue (via DataPower), fetches full claim and settlement data, encrypts the cardholder account number, and writes a normalised WDP event payload into the chbk_outbox_row table for downstream Kafka publishing. After writing, it acknowledges the items as worked so they are removed from the MCM queue.
What it explicitly does NOT do

Does not publish to Kafka directly (despite Kafka dependencies being declared).
Does not handle second/subsequent chargebacks, pre-arbitration, or arbitration stages.
Does not process reversed chargebacks or reversal records.
Does not perform any case routing, decisioning, or response filing.
Does not write to S3 or any file system.
Does not invoke RTSL, RTDS, or any WDP case management REST API to create cases.

Conditional branching on chargebackDetail

Only chargebacks where chargebackType == "CHARGEBACK" AND !reversed AND !reversal are processed.
Other types (REVERSAL records or reversed chargebacks) are excluded at processor step 2.
For multiple qualifying chargebacks in one claim: account number encryption is performed only for the first (isAccountNumberRequired flag flips to false after first iteration).
In the update path (processUpdatedClaims), account number encryption is never performed regardless.

What is unique about first chargeback vs. subsequent
The job only processes chargebackType == "CHARGEBACK" records — not chargebackType == "ARBITRATION" or "SECOND_CHARGEBACK" etc. (though those type strings are inferred from MCM convention; only "CHARGEBACK" is hardcoded in ApplicationConstants.CHARGEBACK).

Dispute stage is hardcoded to "CH1" (First Chargeback) across all records.
PAN encryption and settlement lookup only apply to first-chargeback processing — there is no equivalent flow in this batch for pre-arb or arbitration.
The enrichmentFailure flag on OriginalTransIdentifier is hardcoded to true — signalling to downstream processors that transaction enrichment data was not fully resolved at this stage.


6. PAN HANDLING
Is PAN present in MasterCard responses?
Yes. Two sources:

ClaimDetail.primaryAccountNum — present in the claim detail response from MCM.
SettledTransactionDetail.mastercardMappingServiceAccountNumber — the preferred (mapped) account number from the settled transaction.
SettledTransactionDetail.primaryAccountNumber — also present but not used by the application.

Encryption pipeline

Source: claimDetail.primaryAccountNum or settledTransactionDetail.mastercardMappingServiceAccountNumber (preferred).
Pre-processing: Truncate to last 16 characters if > 16 chars.
Masking check: If the account number contains "*" (already masked), encryption is skipped — the masked value is written as-is.
Encryption call: DisputeServiceImpl.encryptAccountNumber() calls RestInvoker.POST(encryptionUrl, {pan: accountNumber}).
Auth for encryption call: Bearer token from IdpTokenService.getIdpToken() (IDP service).
Result: EncryptionResponse.hpan (HPAN — hashed/encrypted PAN).
Written to outbox: The HPAN is stored in CommonEvent.originalTransIdentifier.accountNumber in the payload JSON column of chbk_outbox_row.

Is clear PAN ever written anywhere?

NOT written to the database. Only the HPAN is stored in the payload.
Not logged: ClaimDetail has @ToString(exclude = "primaryAccountNum") preventing PAN from appearing in log output. SettledTransactionDetail has @ToString(exclude = "mastercardMappingServiceAccountNumber").
The account number field in ProcessorUtil.createCommonEvent() receives either the HPAN (if encryption succeeded) or the potentially clear value (if the account number contains "*" or encryption throws and falls through).
If encryption fails, DisputeServiceImpl throws a ServiceException which propagates up and prevents the entity from being saved.

8. ERROR HANDLING AND FAILURE MODES
Chunk size and transaction boundary
Chunk size: 1 (configured via app.batchProperties.chunk).
Each chunk (single claim) is processed within the wdpTransactionManager (JPA). A failure in the writer for one claim does not roll back previously written claims (separate transactions per chunk).
Retry strategy
CallRetry CountDelayBackoffClassDataPower GET (queue, claim, settlement)32000ms fixedNoneDataPowerRestInvokerDataPower PUT (acknowledge)32000ms fixedNoneDataPowerRestInvokerEncryption POST32000ms fixedNoneRestInvokerIDP token GET32000ms fixedNoneIdpRestInvoker
All retries use spring-retry @Retryable. The retry count "3" and delay "2000" are String constants in ApplicationConstants evaluated as SpEL expressions in maxAttemptsExpression and delayExpression.
No exponential or multiplier backoff is configured.
Whether one record failure stops the batch
No. In BatchItemProcessor.process(), the entire processing block is wrapped in try/catch(Exception ex), which logs the error and returns an empty processedItems list. This means failed claims are silently skipped — the batch continues to the next claim. The failed claim is not acknowledged (since there are no processed items to add to acknowledgeList), so it remains in the MCM queue for the next run.
Dead letter mechanism
None explicitly implemented. There is no DLQ, error table, or separate error-capture mechanism. Failed claims remain in the MCM queue. The error_code and error_message columns in chbk_outbox_row exist in the entity but are never populated by this job.
Job-level failure — restart and checkpoint

Spring Batch's JobRepository stores job execution state in the {prefix}* metadata tables.
With chunk size 1, each claim's outbox write commits independently. On restart after a mid-run failure, Spring Batch would re-execute the step. Because BatchItemReader.searchQueueItems() is @BeforeStep, it re-fetches the queue on every step.
Checkpoint risk: Since acknowledged items are removed from the MCM queue immediately after the writer commits them, a crash between write and acknowledge could result in the outbox row existing but the MCM item still being in the queue — causing a duplicate write attempt on the next run. The idempotency check in the processor (see Section 9) mitigates but does not eliminate this.


9. IDEMPOTENCY
Mechanism
Before writing a new outbox row, the processor checks whether chbk_outbox_row already contains a record for the (networkCaseId, chargebackId) pair:
New claim path (processClaims):
First calls chbkOutboxService.findChbkByNetworkCaseId(claimId) — if this returns any rows, the update path is taken instead.
Update/existing path (processUpdatedClaims):
For each ChargebackDetail, calls chbkOutboxService.chbkOutboxLookup(claimId, chargebackId) which executes:
sqlSELECT COUNT(*) FROM wdp.chbk_outbox_row
WHERE c_ntwk_case_id = ? AND c_ntwk_phase_id = ?
If count > 0, status is set to "SKIPPED".
What "SKIPPED" means
A ProcessedItem with skipCase = true and status = "SKIPPED" is still created and passed to the writer. The writer does not check skipCase before saving — it saves the entity regardless. This means a duplicate run will write a duplicate row with status "SKIPPED" for already-processed chargebacks.

⚠️ Duplicate risk: If the same claim is processed twice (e.g., restart or double-trigger), the first run writes status="PENDING", the second run writes a second row with status="SKIPPED". The downstream publisher must deduplicate on (c_ntwk_case_id) or the "PENDING" status.

The acknowledgement to MCM should prevent re-queuing under normal conditions, but if acknowledgement fails (network error after write), the claim remains in the queue and will be processed again.
PENDING rows from previous runs
There is no mechanism to process stale "PENDING" rows from failed prior runs. The update path only marks new chargebacks within an existing case as "PENDING" — it does not re-evaluate or retry previously-failed "PENDING" rows.

10. VOLUME AND THROUGHPUT
Records per run
Not statically configured. The number of records equals the size of the MCM unworked chargeback queue at poll time. No maximum cap per run is implemented.
Chunk/page size
Chunk size: 1 (one QueueClaim per chunk/transaction). The reader fetches all queue items in a single batch call before the step starts (via @BeforeStep), then returns them one at a time.
Polling frequency
Determined by ${scheduler_cron} — production value unknown (placeholder). The job runs within the same JVM continuously, so frequency is purely a function of the cron expression.
Rate limiting on MasterCard API
Not implemented in this application. No rate-limiting, throttling, or sleep-between-calls logic exists. No MCM rate limit documentation is referenced in the codebase.
Maximum record cap per run
None. All items returned by the queue poll are processed.
