## Executive Summary

`gcp-nap-dispute-event-consumer` is a **stateful, event-driven Spring Boot microservice** that consumes
dispute notification messages from an AWS MSK (Managed Streaming for Kafka) topic and translates them
into actions on a downstream case-management platform (referred to internally as "NAP" – Network Acquirer
Platform). It ingests a single unified `NotificationEvent` payload which encodes three logical event
sub-types (SRV116, WIN/LOSS, SRV118), determines whether an existing dispute case exists, and either
creates a new case or appends an action to an existing one. The service maintains a PostgreSQL error-
tracking database (`NAP.DISPUTE_EVENT_CONSUMER_ERROR`) as a durable dead-letter store, with a companion
REST endpoint that allows manual reprocessing of errored events. It does **not** produce to a Kafka topic;
all outbound communication is HTTP REST to downstream services. Retry logic is implemented via Spring
Retry on every downstream call.

## 1. Primary Responsibility – What It Does and Does NOT Do

### What It Does

- **Consumes** `NotificationEvent` JSON messages from a single AWS MSK Kafka topic using manual offset
  acknowledgement[^1].
- **Classifies** each incoming event into one of three sub-types: `IN_SRV116` (chargeback notification
  from card network), `IN_WIN_LOSS` (arbitration win/loss outcome), or `IN_SRV118` (chargeback response,
  e.g. representment/write-off)[^2].
- **Normalises** the card network name (maps raw scheme names to canonical `CardSchemeName` enum values;
  promotes MASTERCARD + GCMS identifier to MAESTRO)[^3].
- **Orchestrates** a multi-step API pipeline** via a string-valued `apiName` state machine on the
  `NapMessageEvent` wrapper, calling between 3 and 10 downstream REST APIs per event depending on event
  type and case existence[^4].
- **Creates or updates** dispute cases** in the downstream case management system[^5].
- **Persists** error state** to a PostgreSQL table (`NAP.DISPUTE_EVENT_CONSUMER_ERROR`) when any step
  fails, storing the full raw Kafka JSON payload for later reprocessing[^6].
- **Exposes** a REST endpoint** (`POST /merchant/gcp/event-consumer/nap/event`) that allows manual
  reprocessing of a previously-failed `ConsumerError` record[^7].
- **Checks for previous unresolved errors** for the same ARN/NetworkCaseId before processing a new
  event, and blocks or escalates the new event's status if prior errors exist in ERROR state[^8].

### What It Does NOT Do

- Does **not** produce any Kafka messages (there is no `KafkaProducer` or `KafkaTemplate` in the codebase).
- Does **not** handle Visa FraudSwitch lookups (the `FRAUD_SWITCH_LOOKUP` step is commented out and
  skipped).
- Does **not** create the Kafka topics (auto-topic creation is explicitly disabled: 
  `ALLOW_AUTO_CREATE_TOPICS_CONFIG = false`)[^10].
- Does **not** hold any in-memory state between events; each `NotificationEvent` is processed
  independently through the pipeline.
- Does **not** do any inter-consumer coordination — there is no leader election, locking, or
  distributed cache.

### Stateful / Stateless / Persistence

| Dimension | Value |
|---|---|
| Within-event processing | **Stateless** — `NapMessageEvent` is a per-invocation object; all state lives
  in the call stack |
| Across-event / error recovery | **Stateful** — failure state is persisted to PostgreSQL
  (`NAP.DISPUTE_EVENT_CONSUMER_ERROR`) |
| Token caching | **Thread-local stateful** — `KafkaTokenHolder` caches the IDP JWT per thread,
  cleared in a `finally` block after each event[^11] |
| Kafka consumer | **Stateful** — consumer group offset is committed after each message |

## 2. Event Types: Processing Paths

### Event Type Identification

All three event sub-types arrive on the **same Kafka topic** as the same `NotificationEvent` POJO. The
sub-type is determined at runtime inside `CaseServiceImpl.updateEventType()`[^13]:

| Sub-type constant | Identification condition |
|---|---|
| `IN_WIN_LOSS` | `outcome` field is non-blank |
| `IN_SRV118` | Both `napResponseType` AND `napResponseCode` are non-blank (and `outcome` is blank) |
| `IN_SRV116` | All other cases (default) |

**Key decisions**:

### Event Type B: `IN_WIN_LOSS` – Arbitration Win/Loss Outcome

**Processing path**[^16]:

Modify_Card_Network
  → Check_And_Process_Previous_Error_Notification
    → Case_Lookup (GET)
      → Case_Action_Lookup (GET)   [case must exist; outcome present]

**Outbound calls**: Case Search, Case Action Search, Pre-Action Status Rules, Update Case, (optionally)

### `@Recover` Behaviour

Every `@Retryable` method has a paired `@Recover`. On exhaustion of retries, the recover method:
- Sets `notificationEvent.errorOccured = true`.
- If **not** an error-topic reprocessing event: writes a new `ConsumerError` record to PostgreSQL with
  status `FAILED1` (or `SKIPPED_FAILED` for notes/API-log failures, which are non-blocking).
- If **is** an error-topic reprocessing event: sets error fields on `NapMessageEvent` for the caller to
  inspect.
- Sets `apiName = ""` (blank), halting the pipeline[^24].

The offset is acknowledged **before** processing begins[^28]:
```java
// KafkaConsumer.java line 33
```

This is an **at-most-once** delivery guarantee: if the application crashes after acknowledging but before
finishing processing, the event is lost (will not be redelivered). This is a deliberate design choice —
the service manages its own error state via the `DISPUTE_EVENT_CONSUMER_ERROR` table rather than relying
on Kafka redelivery.

### Deserializer and Error Handling on Bad Payload

The value is deserialized using `JsonDeserializer<NotificationEvent>` wrapped in
`ErrorHandlingDeserializer`[^30]:
```java
ErrorHandlingDeserializer<NotificationEvent> errorHandlingDeserializer =
    new ErrorHandlingDeserializer<>(deserializer);
```

If JSON deserialization fails, `ErrorHandlingDeserializer` catches the exception and delivers a `null`
(or a `DeserializationException`-annotated record) rather than crashing the listener. The
`notificationListener` factory is configured with an empty `CommonErrorHandler` (`new
CommonErrorHandler(){}`) [^31], meaning Spring will use its default behaviour (log the error and
continue).

### Dead-Letter Queue (DLQ) / Dead-Letter Behaviour

**There is no Kafka-level DLQ.** The service does not configure `DeadLetterPublishingRecoverer` or any
`@SendTo` dead-letter topic. Instead, it uses a database-backed error queue:

1. On processing failure: a `ConsumerError` row is written to `NAP.DISPUTE_EVENT_CONSUMER_ERROR` with
   the raw Kafka JSON in the `C_KAFKA_EVENT` column and a status of `FAILED1`[^6].
2. The `C_ERROR_REASON` column records which pipeline step (`apiName`) failed and what the exception
   message was.
3. Status escalation lifecycle: `FAILED1` → `FAILED2` → `ERROR` on repeated failures; `DRAFT` for
   blocked-on-prior-error; `SKIPPED` / `SKIPPED_FAILED` for business-rule-skipped events; `PROCESSED`
   on successful reprocessing[^34].
4. Reprocessing is triggered manually via `POST /merchant/gcp/event-consumer/nap/event`.

### Idempotency on Redelivery

Because the offset is acknowledged before processing[^28], **Kafka-level redelivery of the same message
does not occur under normal operation**. However, the service has several idempotency guards for
duplicate processing at the application level:

1. **Duplicate action check**: before creating/updating a case action, `newCaseActionLookup` inspects
   existing action summaries using `NotificationConsumerUtil.caseActionPredicate()` which matches on
   `acquirerCaseNumber` + `uniqueId` + `stageCode` + `actionCode` + `creditDebitIndicator`. If a match
   is found, the event is SKIPPED with a `SKIPPED` error entry[^35].
2. **SRV116 duplicate check**: `isCbkActionAlreadyExists()` checks for an existing action with the same
   source case ID and unique ID before applying a chargeback response[^19].
3. **Previous error gate**: `checkAndProcessConsumerErrorSrvNotification()` queries the error table by
   ARN/NetworkCaseId before processing and blocks the new event if a prior ERROR-status entry
   exists[^8].
4. **Win/Loss NA/CLOSED skip**: if the `outcome` field is `NA` or `CLOSED`, the event is silently
   discarded with a SKIPPED record, preventing unwanted case updates[^16].

# Technical Research Report: Pod Scaling, Architectural Risks & Planned Changes
**Service:** `gcp-nap-dispute-event-consumer` (v1.6.5)

---

## Executive Summary

The service's Kubernetes deployment is a **Deployment** (not a StatefulSet) with replicas driven purely
by a placeholder value — there is no HPA configured in the repository, and CPU limits are entirely
absent from the manifest. The resource manifest exposes several operational risks: no CPU throttle
boundary, no PodDisruptionBudget, and a `WhenUnsatisfiable: ScheduleAnyway` topology spread that
provides no hard availability guarantee. Within the codebase, significant architectural risks include:
at-most-once Kafka delivery with no replay safety net at the broker level, a shared `RestTemplate` bean
with no timeout/connection-pool configuration, a `SEND_BUSINESS_RULES_INFO_TO_KAFKA` pipeline terminal
state that never actually sends to Kafka, a deep recursive error-reprocessing flow vulnerable to stack
overflow on large error queues, and no circuit breaker on any of the 11 downstream REST dependencies.
Evidence of active migration work is visible throughout: a `wdpOnly` / `migrationStatus` flag gates WDP-
only records from processing, commented-out `FRAUD_SWITCH_LOOKUP` and `VISA_CASE_LOOKUP` steps mark
intended-but-deferred functionality, and a commented user-story reference (`US2137854`) documents a
replaced amount-calculation strategy.

## 1. Pod Count and Scaling

### 1.1 Replica Count

The replica count is **fully externalised** via an XL Deploy placeholder:

### 1.2 HPA Configuration

**No HorizontalPodAutoscaler manifest exists** in `resources.yaml` or anywhere else in the
repository[^2]. The service does **not auto-scale based on CPU, memory, or any custom metric**. All
scaling is manual, done by changing the XL Deploy dictionary value for `replicas-mdvs-gcp-nap-dispute-

scales linearly only with pod count, and lag-based autoscaling (e.g., KEDA on consumer group lag) is
not present.

## 2. Architectural Risks and Failure Modes

### Risk 1 – At-Most-Once Kafka Delivery (Severity: HIGH)

The Kafka consumer **acknowledges the offset before processing begins**:
```java
// KafkaConsumer.java:33-41
acknowledgment.acknowledge();              // commit first
NapMessageEvent napMessageEvent = new NapMessageEvent();
...
caseService.processSrv116Notification(napMessageEvent);  // then process
```

- **Failure mode**: If the JVM crashes, pod is OOM-killed, or a network partition occurs after the
  `acknowledge()` call but before `processSrv116Notification()` completes, the event is **permanently
  lost** — Kafka will not redeliver it because the offset has already been committed.
- There is no Kafka DLQ (`DeadLetterPublishingRecoverer` is not configured). The only safety net is
  the PostgreSQL error table, which is only written **during** processing — not before it begins.
- **Severity**: HIGH. Dispute events that are never processed will result in unrecorded chargebacks.

### Risk 2 – No CPU Resource Limit (Severity: MEDIUM)

No CPU request or limit is set in `resources.yaml`[^3]. The Spring Boot JVM with Logstash appender
and extensive JSON serialization/deserialization can spike CPU. Without a limit:
- A noisy-neighbour pod could starve other workloads on the same node.
- Under sustained load, the pod cannot be predictably scheduled or throttled.
- OOM-killer behaviour becomes unpredictable when CPU and memory pressure coincide.

### Risk 3 – Bare RestTemplate with No Timeout Configuration (Severity: HIGH)

`RestTemplate` is instantiated as a no-arg Spring bean with no connection timeout, read timeout, or
connection pool:
```java
// CommonConfig.java:12-13
@Bean
public RestTemplate getRestTemplate() {
    return new RestTemplate();  // no HttpComponentsClientHttpRequestFactory, no timeouts
}
```

- The underlying `SimpleClientHttpRequestFactory` uses JVM defaults (effectively infinite read timeout
  on most platforms).
- The single Kafka consumer thread will block indefinitely if any downstream call (Case Search,
  Workflow Rules, IDP token, etc.) hangs or becomes slow. With `concurrency=1`, this means **the entire
  Kafka consumer pauses** — offset commits stop, the consumer group heartbeat continues but
  `max.poll.interval.ms` will eventually be exceeded, causing a consumer group rebalance and partition
  re-assignment.
- No circuit breaker (Resilience4j, Hystrix) is present anywhere in the codebase (`pom.xml` has no
  resilience4j or spring-cloud-circuitbreaker dependency).
- **Severity**: HIGH. A single slow downstream call can halt all message processing.

### Risk 4 – `SEND_BUSINESS_RULES_INFO_TO_KAFKA` Is a No-Op Terminal State (Severity: MEDIUM)

The constant `ApplicationConstants.SEND_BUSINESS_RULES_INFO_TO_KAFKA = "SEND_BUSINESS_RULES_INFO_TO_
