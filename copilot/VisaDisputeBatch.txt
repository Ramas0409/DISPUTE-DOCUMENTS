## Executive Summary

`gcp-visa-disputes-processor-batch` is a Spring Batch application that polls Visa's RTSI (Real-Time Settlement Interface) batch queues on a cron schedule, processes incoming dispute lifecycle events, enriches them against several internal WDP APIs, and writes structured case events to a PostgreSQL transactional outbox table (`wdp.chbk_outbox_row`) for downstream Kafka publishing. There is exactly **one Spring Batch Job** with **one Step** that processes all seven configured Visa queues sequentially within a single execution. The application is deployed as a **Kubernetes Deployment** (not a CronJob); the embedded `@Scheduled` cron drives execution.

## 1. RESPONSIBILITY

### What the Job Does

This batch application polls Visa's RTSI batch queues for inbound dispute lifecycle events across seven queue types (first chargebacks, pre-arbitrations, arbitrations, acceptances, recalls, pre-compliances, and compliances). For each queued item, it determines whether to create a new WDP dispute case or update an existing one, maps the Visa dispute data to a WDP `CaseEvent`, encrypts the PAN, and persists the result to the `wdp.chbk_outbox_row` transactional outbox. It then calls Visa RTSI to mark the item as read/consumed from the queue for eligible platforms.

### What the Job Explicitly Does NOT Do

- It does **not** process Mastercard disputes (only Visa `networkId` values are filtered in)[^1].
- It does **not** directly publish to Kafka; it writes to a transactional outbox (`chbk_outbox_row`) with status `PENDING` ŌĆö a separate downstream process handles actual Kafka publishing[^2].
- It does **not** send dispute responses or updates back to Visa (only acknowledgement via mark-as-read).
- It does **not** handle merchant-side transaction lookup from a WDP internal database; the transaction lookup service call is commented out in the processor[^3].
- It does **not** manage dispute outcomes (accept/deny) ŌĆö only inbound queue ingestion.

### Single Job vs Multiple Jobs

There is **one Spring Batch Job** (`visaDisputesProcessorJob`) with **one Step** (`processVisaDisputesProcessorBatchStep`). The `BatchItemReader` iterates through all queues in the configured `queueNameList` sequentially within a single job execution ŌĆö not as separate steps or jobs[^4].

### How the Reader Determines Which Queue to Poll

The reader holds a `queueNameList` injected from `props.getBatchProperties().getQueueNameList()` which is populated from the environment variable `${queue_name_list}` in `application.yml`[^12]. On `@BeforeStep`, it sets `nextQueueSeq = 0` and calls `retrieveQueueItems()`. On each `read()` call, it returns items from the current queue. When items are exhausted, it increments `nextQueueSeq` and loads the next queue ŌĆö continuing until all queues are drained[^13].

If a queue returns empty (status code `I-300600000`), the exception causes the reader to advance to the next queue without failing the job[^14].

Step 2: IDP Token Acquisition

For items with networkId = 0002[^26]: Calls visaRTSIService.searchVisaTransaction() to get CardholderIdMethodCode.

If code ŌēĀ "2" and coreDataNeedToMigrate = true ŌåÆ sets platform to CORE, continues.

7. ERROR HANDLING
Retry Strategy
RestInvoker.post() is annotated with Spring Retry[^39]:

java
@Retryable(
  retryFor = {Exception.class},
  maxAttemptsExpression = ApplicationConstants.RETRY_COUNT,  // *3*
  backoff = @Backoff(delayExpression = ApplicationConstants.RETRY_DELAY)  // *2000* ms
)
Max attempts: 3 (1 original + 2 retries)
Delay between retries: 2,000 ms (2 seconds)
Retries on: Any Exception
This applies to all internal WDP service calls. RTSI calls (dataPowerRestInvoker) also go through RestInvoker.post() and benefit from the same retry.

DisputeServiceImpl.transactionLookup() has explicit retry-count awareness: on retryCount < 2, re-throws ServiceException (triggering another retry); on final attempt (count Ōēź 2), throws LoggingException to log and skip[^40].

AOP Error Logging
An @AfterThrowing aspect on LoggingException persists error details to the gcp-api-log-service API log, capturing networkCaseNumber, platform, cardNetwork, and up to 10,000 characters of stack trace[^32].

Item-Level Isolation
Processor: Exceptions are caught per item ŌĆö the processor catches both LoggingException and Exception and returns null, skipping that item. One item failure does not stop the batch[^41].

Writer: The writer iterates the chunk and wraps each item in a try/catch. However, the catch block calls break ŌĆö stopping processing of the remaining items in the current chunk[^42]. Since chunk size = 1 (see below), this is effectively equivalent to skipping that item.

Dead Letter / Error Tracking
LoggingException (non-empty flag) ŌĆö persisted to gcp-api-log-service via AOP aspect.
Writer duplicate-check failures ŌĆö ChbkOutboxEntity row with status = ERROR written to database.
No dedicated DLQ or dead letter queue mechanism; failed items (returning null from processor) are simply not written.

### Duplicate Check in Writer

The writer performs a database lookup before saving any non-RECALL queue item[^43]:
```java

  caseEvent.getNetworkCaseId(),

);
```

The JPQL query:
```sql
SELECT C FROM ChbkOutboxEntity C
WHERE C.networkCaseId = :networkCaseId
  AND C.networkPhaseId = :networkPhaseId
  AND C.caseStage = :disputeStage
```

If a match is found, the event is saved with `status = SKIPPED` instead of `PENDING`, and the item is still marked as read on Visa RTSI.

### RECALL Queue Bypass

The RECALL queue **explicitly skips the duplicate check**[^9]:
```java
if (!QueueName.RECALL.getValue().equalsIgnoreCase(item.getQueueName())) {
  // duplicate check...
}
```

No idempotency protection for RECALL queue items.

### Missing Data Path

If `networkPhaseId` or `networkCaseId` or `disputeStage` is blank, the duplicate check cannot be performed. The writer saves the row with `status = ERROR` and a log message "Unable to perform duplicate check due to missing data."[^44]

### Queue-Side Deduplication Gaps

- If the job is restarted before `markAsRead` is called (e.g., pod crash), items not yet marked as read will be re-polled from Visa RTSI.

## 3. CONCURRENCY AND ORDERING

### Sequential vs Parallel Queue Processing

All queues are processed **strictly sequentially in a single thread**. The `BatchItemReader` holds a pointer (`nextQueueSeq`) and processes queue N to exhaustion before advancing to queue N+1[^9]. There is no multi-threading, `TaskExecutor`, or parallel step configuration in `JobConfiguration`[^10].
```java
// BatchItemReader.java ŌĆö single-threaded sequential iteration
@BeforeStep
public void searchQueueItems(StepExecution stepExecution) throws Exception {
  nextQueueSeq = 0;
  queueNameList = props.getBatchProperties().getQueueNameList();
  queueListSize = queueNameList.size();
  retrieveQueueItems(); // loads queue[0]
}

public QueueReaderItem read() throws Exception {
  if (recordIndex >= queueItemsCount) {
    // returns current item
    if (recordIndex == queueItemsCount) {
      nextQueueSeq++;        // advance to next queue
      retrieveQueueItems();  // load next queue
    }
    return readerItem;
  }
  return null; // step ends
}
```

### Ordering Within a Queue

Items within a queue are processed **in the order returned by the Visa RTSI API**. No sorting or re-ordering is applied by this application. Whether Visa RTSI guarantees ordering (e.g., by `DateToRespond` or `BatchQueueItemSID`) is not determinable from the source alone ŌĆö the application makes no assumption about order.

**Ordering does not matter** for processing correctness: each item is independently processed via case lookup (create or update) with no dependency on prior items in the same queue run.

### Partition or Merchant-Level Isolation

**No partitioning or merchant-level isolation exists.** All items from all queues are processed in a flat sequential pipeline regardless of merchant, acquirer BIN, or geographic region. There are no `Partitioner` implementations and no merchant-specific threading[^10].
