## Purpose of This Document

This is the master database reference for the WDP platform. It contains:

1. **Database Instances** — all database systems WDP connects to and
   who owns them
2. **Schema Ownership Map** — every schema and table, which component
   owns it (writes to it), and which components access it read-only
3. **Shared Table Risk Register** — tables written by more than one
   component, flagged as coordination risks
4. **Data Retention and Compliance Notes**

Individual component files (WDP-COMP-[NN]-*.md) contain the Database
Ownership section specific to each component. This file is the
cross-cutting view — the complete data ownership map for the platform.

> ⚠️ **Schema-DDL caveat (platform-wide):** No component repository
> contains Flyway / Liquibase / `schema.sql` / `data.sql`. All `ddl-auto`
> settings are `false` and all `spring.batch.jdbc.initialize-schema`
> settings are `never`. DDL — including UNIQUE constraints, indexes, and
> column existence on shared tables — is managed in a schema-owning
> repository outside the component repos. **Unique-constraint
> enforcement on every shared table below cannot be confirmed from the
> component sources alone.** DBA confirmation is required for every
> `(idempotency_id, channel_type, event_timestamp)`,
> `(file_name, s3_key)`, `(file_job_id, event_type, row_number)`,
> `(c_ntwk_case_id, c_ntwk_phase_id)`, and `(I_CASE, I_ACTION_SEQ)`
> uniqueness assumption in this document.

**`wdp.chbk_outbox_row` — writer-specific notes:**

- **`created_by` / `updated_by` constants — four distinct values across batch writers:**
  - COMP-07 VisaDisputeBatch: `"WVDPB"`
  - COMP-08 FirstChargebackBatch: `"WMFDPB"`
  - COMP-09 CaseFillingBatch: `"WMPAPB"`
  - COMP-11 FileProcessor: `"FILE_PROCESSOR"` *(corrected 2026-04-18 from previously documented `"WPFLEPR"`)*
- **No DB unique constraint confirmed on this table** — no DDL / Flyway / Liquibase / entity-level `uniqueConstraints` in COMP-07, COMP-08, COMP-11, COMP-13, or COMP-23 repos. `ddl-auto: false`; `initialize-schema: never`. Whether a unique index exists in the live schema must be confirmed with the DBA team.
- **COMP-09 update path always INSERTs a new row** — existing rows are never UPDATEd by COMP-09. A claim that progresses through its phase lifecycle produces multiple rows, one per phase observation, distinguished by status (PENDING / SKIPPED). Table growth is proportional to phase count × rerun frequency.
- **COMP-09 writer swallows all save exceptions** — DB save failure produces no error row, no retry, and no Spring Batch skip counter increment; the chunk reports success. Same pattern observed on COMP-07's `BatchItemWriter`. Cross-batch platform pattern.
- **COMP-09 skip paths write no row at all** — claims that fail null-field validation, stage-determination, IDP token, or encryption leave zero database trace. `error_code` / `error_message` columns exist on the entity but are never populated by COMP-09.
- **COMP-09 c_case_stage concatenation** — base stage (`PAB`, `ARB`, `PRA`, `AII`, `AIM`) is written EXCEPT when `issuerCaseStatus = withdraw`, where the column is written as `{stage}_withdraw` (e.g. `PAB_withdraw`, `PRA_withdraw`). Downstream consumers that key on `c_case_stage` must tolerate this concatenation.
- **COMP-09 `network_notes` column** — written conditionally, only when `schemeRef.notes` is non-empty. No other batch writer populates this column.
- **COMP-09 `c_ntwk_case_id` semantics** — value is `standardClaimId`, which equals `claimDetail.standardClaims` when `claimType = "CaseFiling"`, else `claimId`. The DRAFT label `networkCaseId` is loose.
- **COMP-08 duplicate-check writes SKIPPED marker rows rather than suppressing the insert** — `skipCase` flag on `ProcessedItem` is computed but never consumed by its Writer. One SKIPPED row accumulates per re-polled known chargeback per COMP-08 scheduler run. Growth boundedness depends on whether COMP-12 Scheduler2 archives SKIPPED rows — not currently confirmed.
- **COMP-08 update-path writes status=PENDING with accountNumber=null** on newly-arrived chargebacks on an existing claim — `processUpdatedClaims` never sets `isAccountNumberRequired=true`. Suspected defect.
- **COMP-08 writer-ACK hazard** — mid-chunk JPA save failure does not prevent the subsequent ACK PUT to MCM; ACK PUT exceptions are swallowed silently with no log, no metric, no outbox marker. Items acknowledged off MCM with no corresponding outbox row are unrecoverable.
- **COMP-07 `skipCase` items write no outbox row at all** (previously implied SKIPPED was written — corrected).
- **COMP-11 entity in this repo does not declare `kafka_topic` / `kafka_partition` / `kafka_offset` / `idempotency_id` / `retry_count` / `next_retry_at` / `published_at`** — consistent with COMP-12 being sole setter of these columns.
- **COMP-11 initial CHARGEBACK_PROCESS status is PENDING (PrePersist).** LOADING is transient during the evidence phase only (parent flipped by `insertEvidenceOutBox @Transactional`). DCPO evidence rows are explicitly BLOCKED; DCPO parent stays PENDING. DNWK writes CHARGEBACK_PROCESS at PENDING directly with no evidence phase.
- **COMP-12 idempotencyId is typed as `UUID` on this table's entity**, in contrast to `wdp.outgoing_event_outbox` (String) — cross-table type inconsistency to flag for any downstream consumer doing shared dedup logic.
- **COMP-23 `findById(...).isPresent()` UPDATE guard is no-op if the row is missing.** No terminal-status guard — a PROCESSED row can be overwritten. Same `@Transactional` as the case save.
- **SUCCESS rows are archived to `wdp.chbk_outbox_row_archive` by COMP-12 Scheduler2 after 30 days.**

**`wdp.outgoing_event_outbox` — writer-specific notes:**

- 🔴 **Correction 2026-04-18:** COMP-18 NotificationOrchestrator was previously documented as a writer of this table. Source grep confirms zero references in the `wdp-outgoing-consumer` repo. **COMP-18 has been removed from the writer list.** COMP-18's writes land exclusively on `wdp.bre_orchestration_outbox` and `wdp.file_generation_event` (see rows below).
- **COMP-43 lifecycle detail:**
  - `created_by` / `updated_by` hardcoded to `"PCSECRTC"` on every write
  - `next_retry_at` is set on every non-SUCCESS update (not only on FAILED)
  - `retry_count` is incremented only on incoming `status==FAILED`; ERROR escalation fires on the third such write
  - `original_event` column carries the full `OutgoingEvent` JSON, including fields not used by COMP-43 (`actionStatus`, `expirationDate`, `responseDueDate`, `dateReceivedByAcquirer`, `level1Entity`–`level5Entity`, `disputeStage`, `channelType`)
  - ⚠️ **No Kafka-metadata write-back columns** (`kafka_topic`, `kafka_partition`, `kafka_offset` not present on the entity) — incident correlation between consumer logs and outbox rows requires log-side join
  - Idempotency check SELECT and INSERT run in **separate JPA short transactions** (`processNewCaseActionEvent` not `@Transactional`); race window is wider than v1.0 implied
- **COMP-17 lifecycle detail:**
  - No service-level `@Transactional` on `OutgoingEventOutboxServiceImpl` — every save runs in the implicit per-call JPA transaction
  - Predecessor candidate filter excludes `SUCCESS` and `SKIPPED`
  - Predecessor scope is `caseNumber + channelType=EXPIRY_EVENTS + id < currentId` only — **not scoped on `i_action_seq`**, causing cross-action interference (a stuck `ERROR`/`FAILED` row for action A on case X blocks all subsequent EXPIRY_EVENTS for the same case regardless of `action_seq`)
  - Status mapping: any non-ERROR predecessor (FAILED, PENDING, PENDING_DEFERRED, BLOCKED, PUBLISHED) maps the current row to PENDING_DEFERRED
- **COMP-41 lifecycle detail:**
  - Writes via independent auto-commit JPA saves — zero `@Transactional` annotations across `src/`. SELECT-before-INSERT duplicate check is therefore **not atomic**.
  - `idempotency_id` typed as `String` on its entity — same as COMP-17, COMP-43; cross-table type inconsistency with `chbk_outbox_row` (UUID) confirmed for COMP-41 too
  - **Three distinct PUBLISHED-orphan paths**, all unrecoverable in-component, all invisible to COMP-12 Scheduler3 if Scheduler3 reads only FAILED / PENDING_DEFERRED: (a) post-ACK crash before Signifyd response; (b) Signifyd empty body ("NO_DATA_FROM_SIGNIFYD") — no status transition; (c) final outbox UPDATE failure after ACK
- **No DB UNIQUE constraint visible in any of COMP-17, COMP-41, or COMP-43 repos** — DBA confirmation required for `(idempotency_id, channel_type, event_timestamp)` enforcement.
